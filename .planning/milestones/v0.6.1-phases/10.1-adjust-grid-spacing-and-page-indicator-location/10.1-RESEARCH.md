# Phase 10.1: Adjust Grid Spacing and Page Indicator Location - Research

**Researched:** 2026-03-14
**Domain:** QML layout math, navbar composition, grid cell sizing
**Confidence:** HIGH

## Summary

This phase addresses three tightly scoped QML changes: (1) relocating page indicator dots from HomeMenu into the navbar flanking the clock, (2) adding an auto-snap threshold to the grid frame computation to recover wasted gutter space, and (3) tightening internal card padding for less visual gap between widgets. All changes are QML-only -- no C++ modifications needed.

The grid math is well-understood. On the target 1024x600 display with bottom navbar, the current grid is 7x3 with 82% Y-axis waste. The 60% auto-snap threshold correctly triggers an extra row, producing a 7x4 grid with cells shrinking ~4.5% (131.9px to 126.0px). Removing the PageIndicator from HomeMenu reclaims ~18px of vertical space but doesn't change the grid count -- the snap threshold is the real fix.

The navbar dot integration is straightforward but requires awareness of all four edge orientations, the horizontal/vertical layout duality in Navbar.qml, and the visibility guard (dots only when home screen is active). The `WidgetGridModel` already exposes `pageCount` and `activePage` as Q_PROPERTYs -- the navbar can bind directly via QML without any new C++ plumbing.

**Primary recommendation:** Implement as three sequential changes: (1) auto-snap threshold in HomeMenu.qml grid computation, (2) delete PageIndicator + add navbar dots, (3) reduce card padding. Each is independently testable.

### Verified Math: 1024x600 Bottom Navbar

| Property | Current | After Snap |
|----------|---------|------------|
| Available (after navbar+margins) | 984x504 | 984x504 |
| DisplayInfo.cellSide (diag/9) | 131.9px | 131.9px (unchanged) |
| gridCols x gridRows | 7x3 | 7x4 |
| X waste | 60.9px (46%) | absorbed into smaller cells |
| Y waste | 108.4px (82%) | absorbed into smaller cells |
| Effective cellSide | 131.9px | 126.0px |
| Cell size reduction | -- | 4.5% |

The snap triggers on Y-axis only (82% > 60%). X-axis at 46% stays below threshold. After snap, effective cell = `min(984/7, 504/4)` = `min(140.6, 126.0)` = 126.0px. Cells remain square.

### Navbar Dot Integration Points

The center control (controlIndex=1) in Navbar.qml currently gets 60% of the bar's length:
- **Horizontal (top/bottom):** `hControl1.width = parent.width * 0.60`, ~614px on 1024w
- **Vertical (left/right):** `vControl1.height = parent.height * 0.60`, ~360px on 600h

The clock text takes maybe 60-80px horizontally. There's ample room for dots flanking it. The clock font size is `Math.round(root.height * 0.75)` in horizontal mode.

### Pattern: Snap-Aware Grid Computation

```javascript
// Recommended implementation for HomeMenu.qml lines 21-28
readonly property real baseCellSide: DisplayInfo ? DisplayInfo.cellSide : 120
readonly property real kSnapThreshold: 0.6

// Initial col/row count from base cell size
readonly property int _baseCols: pageView.width > 0 ? Math.max(3, Math.floor(pageView.width / baseCellSide)) : 3
readonly property int _baseRows: pageView.height > 0 ? Math.max(2, Math.floor(pageView.height / baseCellSide)) : 2

// Snap: if waste exceeds threshold, add a row/col
readonly property real _xWaste: pageView.width - _baseCols * baseCellSide
readonly property real _yWaste: pageView.height - _baseRows * baseCellSide
readonly property int gridCols: _xWaste > kSnapThreshold * baseCellSide ? _baseCols + 1 : _baseCols
readonly property int gridRows: _yWaste > kSnapThreshold * baseCellSide ? _baseRows + 1 : _baseRows

// Effective cell size (square, fit both axes)
readonly property real cellSide: Math.min(pageView.width / gridCols, pageView.height / gridRows)
readonly property real gridW: gridCols * cellSide
readonly property real gridH: gridRows * cellSide
readonly property real offsetX: (pageView.width - gridW) / 2
readonly property real offsetY: (pageView.height - gridH) / 2
```

**Key detail:** `cellSide` is now derived (effective), not raw from DisplayInfo. All downstream references (`model.column * homeScreen.cellSide`, `offsetX`, etc.) already use `homeScreen.cellSide` so they pick up the snapped value automatically.

### Visibility Guard

The navbar is global (Shell.qml:60). Dots must be hidden when not on the home screen. Three conditions define "home screen active":
1. `!PluginModel.activePluginId` -- no plugin active (AA, BT Audio, Phone)
2. `ApplicationController.currentApplication !== 6` -- not settings (6 = settings index)
3. `WidgetGridModel.pageCount > 1` -- more than 1 page exists

All three are already Q_PROPERTYs with NOTIFY signals, so QML binding works directly.

### Anti-Patterns to Avoid
- **Don't modify DisplayInfo.cpp** -- snap logic is QML-only per CONTEXT.md. DisplayInfo.cellSide stays as the baseline.
- **Don't use `pageView.currentIndex` in the navbar** -- the navbar doesn't have visibility of `pageView` (it's inside HomeMenu). Use `WidgetGridModel.activePage` instead, which is kept in sync by HomeMenu's `onCurrentIndexChanged`.
- **Don't introduce inter-cell gap math** -- pitch == cellSide throughout. Widget spacing is visual only (inner card margin).
- **Don't make dots tappable** -- intentional regression for car-speed usability.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Page state communication | New signal/slot plumbing | Direct QML binding to `WidgetGridModel.activePage`/`pageCount` | Already Q_PROPERTYs with NOTIFY, no C++ needed |
| Dot visibility | Complex state machine | Compound QML binding on 3 existing properties | All three conditions already have change notifications |
| Grid snap math | C++ DisplayInfo changes | QML computed properties in HomeMenu.qml | QML knows the actual available rect, C++ doesn't |

## Common Pitfalls

### Pitfall 1: Double navbar deduction
**What goes wrong:** Subtracting navbar thickness from available grid height in HomeMenu.qml
**Why it happens:** Intuitive to account for the navbar, but Shell.qml ColumnLayout already deducts it via `anchors.topMargin`/`anchors.bottomMargin`
**How to avoid:** HomeMenu's `pageView.width`/`pageView.height` already reflects post-navbar space. Use these directly.
**Warning signs:** Grid shrinks far too much, or changes size when navbar edge changes.

### Pitfall 2: cellSide binding loop
**What goes wrong:** `cellSide` depends on `gridCols`, `gridCols` depends on `cellSide` -- QML binding loop warning
**Why it happens:** If snap-aware code references `cellSide` to compute `gridCols` and vice versa
**How to avoid:** Use `baseCellSide` (from DisplayInfo) for the col/row count computation, then derive effective `cellSide` from the final grid dims. Two-stage computation, no circularity.
**Warning signs:** QML "Binding loop detected" console warnings

### Pitfall 3: Grid dimension change triggers remap cascade
**What goes wrong:** Changing gridCols/gridRows triggers `WidgetGridModel.setGridDimensions()` which remaps all widget placements, potentially reshuffling the user's layout
**Why it happens:** The snap threshold changes the grid from 7x3 to 7x4 -- this IS a dimension change
**How to avoid:** This is expected and correct behavior. The remap system (base snapshot + spiral nudge + spillover) already handles dimension changes gracefully. Existing YAML configs will auto-migrate. The grid_version field (GL-03) handles this.
**Warning signs:** Widgets move to unexpected positions -- but remap is tested and designed for this.

### Pitfall 4: Vertical navbar dot orientation
**What goes wrong:** Dots render horizontally even when navbar is in vertical (left/right) mode
**Why it happens:** Using a Row layout without checking `isVertical`
**How to avoid:** Use `isVertical` to switch between Row (horizontal) and Column (vertical) layouts, same pattern as the existing horizontal/vertical clock split in NavbarControl.qml (lines 85-112).

### Pitfall 5: pageView reference from navbar scope
**What goes wrong:** Trying to reference `pageView.currentIndex` from Navbar.qml
**Why it happens:** `pageView` is defined inside HomeMenu.qml -- it's not in the navbar's QML scope
**How to avoid:** Use `WidgetGridModel.activePage` which is a context property exposed to all QML. HomeMenu sets it on line 119: `WidgetGridModel.activePage = currentIndex`.

### Pitfall 6: PageIndicator removal breaks ColumnLayout spacing
**What goes wrong:** After removing PageIndicator, the ColumnLayout `spacing: UiMetrics.spacing` (8px) still applies between the SwipeView and... nothing. No issue -- ColumnLayout spacing only applies between visible children.
**How to avoid:** Non-issue. Just delete the PageIndicator block and the SwipeView will expand to fill.

## Code Examples

### Card Padding Reduction (Claude's Discretion)

Current padding (HomeMenu.qml line 280):
```javascript
anchors.margins: UiMetrics.spacing / 2  // ~4px at default scale
```

Recommendation: Reduce to `UiMetrics.spacing / 4` (~2px). This tightens the visual gap between widget cards from ~8px total (4+4 between adjacent cells) to ~4px total (2+2), making the grid feel more cohesive while still maintaining a visible seam between cards. The glass card `radius: UiMetrics.radius` (~10px) needs at least 1-2px margin to avoid clipping artifacts.

```javascript
// Recommended: tighter padding
anchors.margins: Math.max(UiMetrics.spacing / 4, 2)  // ~2px, minimum 2px
```

### Dot-Clock Integration Layout (Horizontal)

The center NavbarControl gets 60% width. Inside it, arrange dots + clock in a Row:

```javascript
Row {
    anchors.centerIn: parent
    spacing: UiMetrics.spacing
    visible: root.showClock

    // Left dots (pages before current)
    Repeater {
        model: dotData.leftCount
        Rectangle {
            width: dotSize; height: dotSize; radius: dotSize / 2
            color: ThemeService.onSurfaceVariant; opacity: 0.4
            anchors.verticalCenter: parent.verticalCenter
        }
    }

    // Clock (IS the active page indicator)
    Text {
        id: clockHoriz
        color: navbar.barFg
        font.pixelSize: Math.round(root.height * 0.75)
        font.weight: Font.DemiBold
        anchors.verticalCenter: parent.verticalCenter
    }

    // Right dots (pages after current)
    Repeater {
        model: dotData.rightCount
        Rectangle {
            width: dotSize; height: dotSize; radius: dotSize / 2
            color: ThemeService.onSurfaceVariant; opacity: 0.4
            anchors.verticalCenter: parent.verticalCenter
        }
    }
}
```

Where `dotData` is computed from `WidgetGridModel.activePage` and `WidgetGridModel.pageCount`:
```javascript
readonly property var dotData: {
    var total = WidgetGridModel.pageCount
    var current = WidgetGridModel.activePage
    return { leftCount: current, rightCount: Math.max(0, total - current - 1) }
}
```

### Snap Threshold as QML Constant

Recommendation: Use a `readonly property` at the top of HomeMenu.qml, not configurable. The 0.6 value was chosen analytically and doesn't need user tuning. Making it configurable adds complexity (YAML key, settings UI, validation) for no real benefit.

```javascript
readonly property real kSnapThreshold: 0.6  // Add row/col when waste > 60% of cell
```

## Open Questions

1. **Card padding value -- how tight is too tight?**
   - What we know: Current is `UiMetrics.spacing / 2` (~4px). Reducing to `spacing / 4` (~2px) halves the visual gap.
   - What's unclear: Whether 2px is enough for the glass card radius (10px) to render cleanly without sub-pixel artifacts on the Pi's 1024x600 display.
   - Recommendation: Start with `spacing / 4`, verify on Pi. Can always adjust.

2. **Dot component placement -- inline or separate file?**
   - What we know: NavbarControl.qml is 122 lines, relatively clean. Adding dots inline would add ~40 lines.
   - What's unclear: Whether future phases will want to reuse the dot indicator elsewhere.
   - Recommendation: Inline in NavbarControl.qml. The dots are tightly coupled to the clock-as-active-indicator concept. Extract later if reuse emerges (YAGNI).
