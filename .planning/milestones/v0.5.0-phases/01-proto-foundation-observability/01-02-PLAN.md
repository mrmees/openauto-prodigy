  <files>src/core/aa/AndroidAutoOrchestrator.cpp</files>
  <action>
The existing channel handlers in open-androidauto already have `default:` cases that log unknown messages, but they use raw `qWarning()`/`qInfo()` (not the categorized `qCDebug(lcAA)` pattern). Modifying the library handlers is possible but heavy-handed. Instead, add observability at the Prodigy orchestrator layer.

Two approaches (Claude's discretion per CONTEXT.md — choose the cleaner one):

**Option A: Signal-based approach** — If the open-androidauto handlers emit an `unhandledMessage` signal from their `default:` case, connect to it in the orchestrator. Check if `IChannelHandler` or specific handlers have such a signal. If not, this won't work without modifying the library.

**Option B: Catch-all via AASession callback** — The AASession already logs unregistered channels at line 320-325 with plain `qDebug()`. The key observability gap is:
1. Messages arriving on channels with NO registered handler (AASession line 320-325) — currently logged with plain `qDebug()`, should use `qCDebug(lcAA)` with hex payload
2. Messages arriving on registered handlers that hit the `default:` case — already logged by the handlers themselves with `qWarning()`

For approach:
- Add a `setUnhandledMessageCallback` or equivalent on AASession (if the API supports it), OR
- Since the library handlers already log their own defaults, focus on the gap: the AASession unregistered-channel path. This is in the library code at `libs/open-androidauto/src/Session/AASession.cpp` line 320-325.

**Concrete implementation:**
1. In `libs/open-androidauto/src/Session/AASession.cpp`, enhance the unregistered channel log (line 320-325) to include hex payload (currently only logs channel ID and message ID):
   ```cpp
   qDebug() << "[AASession] Unhandled: ch" << channelId
            << "msgId" << QString("0x%1").arg(messageId, 4, 16, QChar('0'))
            << "len" << dataSize
            << "hex:" << payload.mid(dataOffset, qMin(dataSize, 128)).toHex(' ');
   ```

2. In `AndroidAutoOrchestrator.cpp`, after registering all known channels (around line 259), add a comment noting which ChannelId values are intentionally unregistered (so future developers know what channels might produce these log lines).

3. The existing handler `default:` cases already provide per-handler logging for unknown message IDs within handled channels. No additional wiring needed there — they're already noisy enough at `qWarning` level.

Log format should invite exploration per user decision: message ID, channel, raw payload — enough to go look it up, not a decoded breakdown. Use `qCDebug(lcAA)` pattern where possible so logs are silent unless verbose AA logging is enabled. For the AASession change (which is in the library, not under lcAA category), use `qDebug()` with a distinctive prefix like `[AA:unhandled]`.
