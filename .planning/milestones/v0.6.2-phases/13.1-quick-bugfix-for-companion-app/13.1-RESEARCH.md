# Phase 13.1: Quick Bugfix for Companion App - Research

**Researched:** 2026-03-15
**Domain:** Qt TCP socket lifecycle, QTimer inactivity detection
**Confidence:** HIGH

## Summary

This is a surgical bugfix to `CompanionListenerService` (single .cpp/.hpp pair). The bug: when a companion app reconnects (e.g., phone WiFi toggle, app restart), the Pi-side service still holds a stale `client_` pointer from the previous connection. `onNewConnection()` rejects the new connection with "already connected" and no log line. The stale socket never fires `disconnected` (TCP half-open), so the service is permanently stuck.

The fix involves four changes: (1) always-replace on new connection instead of reject, (2) connect `QTcpSocket::errorOccurred` signal, (3) add 30s inactivity QTimer, (4) extract shared `clearClientSession()` from the existing `onClientDisconnected()` body. All four converge on the same idempotent cleanup path.

**Primary recommendation:** Extract `clearClientSession()` first, then wire all three new triggers (replace, error, timeout) to call it. Use `QTimer::singleShot` pattern or a member `QTimer*` reset on each valid status message.

### Pattern: Idempotent Cleanup Method

The core pattern for this fix. The cleanup method must be safe to call multiple times in any order:

```cpp
void CompanionListenerService::clearClientSession()
{
    // Guard: already cleared
    if (!client_) return;

    // Disconnect signals FIRST to prevent re-entrant calls
    client_->disconnect(this);

    // Teardown socket
    client_->abort();        // immediate close, no graceful shutdown
    client_->deleteLater();
    client_ = nullptr;

    // Clear session state
    sessionKey_.clear();
    lastSeq_ = -1;

    // Stop inactivity timer
    inactivityTimer_->stop();

    // Clear all published state
    gpsLat_ = 0.0; gpsLon_ = 0.0; gpsSpeed_ = 0.0;
    gpsAccuracy_ = 0.0; gpsBearing_ = 0.0; gpsAgeMs_ = -1;
    phoneBattery_ = -1; phoneCharging_ = false;
    internetAvailable_ = false; proxyAddress_.clear();
    requestedProxyHost_.clear();
    requestedProxyPort_ = 0;
    proxyRouteApplied_ = false;
    pendingWallpaperData_.clear();
    pendingThemeJson_ = QJsonObject();
    expectedWallpaperChunks_ = 0;
    receivedWallpaperChunks_ = 0;

    if (systemClient_)
        systemClient_->setProxyRoute(false);

    emit connectedChanged();
    emit gpsChanged();
    emit batteryChanged();
    emit internetChanged();
}
```

**Critical detail:** `client_->disconnect(this)` MUST come before `client_->abort()`. The `abort()` call may synchronously emit `disconnected` and `errorOccurred`, which would re-enter `clearClientSession()` without the guard. Disconnecting signals first prevents this.

### Pattern: Always-Replace on New Connection

```cpp
void CompanionListenerService::onNewConnection()
{
    auto* pending = server_->nextPendingConnection();

    if (client_) {
        qCWarning(lcCore) << "Companion: replacing stale client"
                          << client_->peerAddress().toString() << ":" << client_->peerPort()
                          << "with" << pending->peerAddress().toString() << ":" << pending->peerPort()
                          << "reason=replaced_by_new_connection";
        clearClientSession();  // tears down old client
    }

    client_ = pending;
    connect(client_, &QTcpSocket::readyRead, ...);
    connect(client_, &QTcpSocket::disconnected, ...);
    connect(client_, &QAbstractSocket::errorOccurred, ...);

    inactivityTimer_->start();  // start/restart 30s timer
    sendChallenge();
}
```

### Pattern: QTimer for Inactivity

**Recommendation: Member `QTimer*` (not singleShot).** A member timer is restartable with `timer->start()` which resets the countdown. `QTimer::singleShot` would require creating and tracking a new timer each time.

```cpp
// In constructor:
inactivityTimer_ = new QTimer(this);
inactivityTimer_->setSingleShot(true);
inactivityTimer_->setInterval(30000);
connect(inactivityTimer_, &QTimer::timeout, this, [this]() {
    if (client_) {
        qCWarning(lcCore) << "Companion: inactivity timeout (30s), disconnecting"
                          << client_->peerAddress().toString() << ":" << client_->peerPort();
        clearClientSession();
    }
});
```

Reset in `handleStatus()` after MAC verification succeeds (line 311 area):
```cpp
inactivityTimer_->start();  // restart 30s countdown
```

### Anti-Patterns to Avoid

- **Calling `deleteLater()` without nulling `client_`:** The pointer remains valid for the current event loop iteration, but signal handlers may fire before deletion. Always null immediately after `deleteLater()`.
- **Using `disconnectFromHost()` in cleanup:** This attempts a graceful TCP close, which may block or trigger additional signals. Use `abort()` for immediate teardown.
- **Connecting errorOccurred without disconnecting signals in cleanup:** `abort()` can trigger `errorOccurred(RemoteHostClosedError)` synchronously, causing re-entrant cleanup. Must `disconnect(this)` first.
- **Starting the inactivity timer before authentication:** A connection that never authenticates would trigger a 30s timeout and cleanup of a socket that was never fully set up. Start the timer on new connection (guards against zombie unauthenticated connections too -- this is actually fine behavior).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Timer mechanism | Custom timestamp comparison in event loop | `QTimer` with `setSingleShot(true)` | QTimer integrates with Qt event loop, handles thread safety |
| Socket error detection | TCP keepalive polling | `QAbstractSocket::errorOccurred` signal | Qt already monitors socket state |

## Common Pitfalls

### Pitfall 1: Signal Re-Entrancy During Cleanup
**What goes wrong:** `abort()` synchronously emits `errorOccurred` and/or `disconnected`, re-entering the cleanup path.
**Why it happens:** Qt signal emission is synchronous by default (direct connection, same thread).
**How to avoid:** Call `client_->disconnect(this)` BEFORE `client_->abort()`. The nullptr guard at the top of `clearClientSession()` is a second line of defense.
**Warning signs:** Double-free crash, "pure virtual method called" abort, or cleanup signals emitted twice.

### Pitfall 2: errorOccurred + disconnected Double-Fire
**What goes wrong:** Some socket errors (e.g., `RemoteHostClosedError`) emit BOTH `errorOccurred` AND `disconnected` in sequence.
**Why it happens:** Qt fires `errorOccurred` first, then `disconnected`. Both are connected to cleanup.
**How to avoid:** The `if (!client_) return;` idempotency guard handles this cleanly. Second call sees nullptr and returns.

### Pitfall 3: Timer Firing After Socket Already Cleaned Up
**What goes wrong:** Timer fires its timeout lambda, but `client_` was already cleaned up by disconnect/error.
**Why it happens:** Timer was started but not stopped during cleanup.
**How to avoid:** `clearClientSession()` MUST call `inactivityTimer_->stop()`. The nullptr guard in the lambda is belt-and-suspenders.

### Pitfall 4: Stale QTcpSocket* After deleteLater()
**What goes wrong:** Code accesses `client_` between `deleteLater()` and actual deletion.
**Why it happens:** `deleteLater()` defers deletion to the next event loop iteration.
**How to avoid:** Set `client_ = nullptr` immediately after `deleteLater()`. All subsequent code checks `client_` before use (already the pattern in `onClientReadyRead()`).

```cpp
                      << (client_ ? client_->errorString() : "no client");
    clearClientSession();
});
```

Note: Use `QAbstractSocket::errorOccurred` (not `QTcpSocket::errorOccurred`) -- the signal is defined on the base class. Both work due to inheritance, but `QAbstractSocket` is the canonical form.
