# Phase 13.2: Fix Saving of Themes Received by Companion App - Research

**Researched:** 2026-03-15
**Domain:** ThemeService config persistence + QML settings UX
**Confidence:** HIGH

## Summary

This is a straightforward bugfix + UX improvement phase touching two files primarily: `ThemeService.cpp/.hpp` (C++ persistence fix) and `ThemeSettings.qml` (UI restructuring). The bug is clear: `ThemeService::setTheme()` loads a theme into memory but never persists the theme ID to `config.yaml`, so any theme switch (including companion imports) reverts on restart. The fix follows an established pattern already used by other services.

The UX changes (wallpaper toggle gate, delete button relocation) are pure QML work using existing components (`SettingsToggle`, `FullScreenPicker`). No new libraries, no architectural changes, no new dependencies.

**Primary recommendation:** Wire `IConfigService*` into ThemeService (same pattern as BluetoothManager), call `setValue` + `save` in `setTheme()`, then restructure ThemeSettings.qml with a toggle-gated wallpaper picker and relocated delete button.

## Architecture Patterns

### Pattern 1: Service IConfigService Injection
**What:** Services that need config persistence take an `IConfigService*` via setter or constructor, then call `setValue()` + `save()`.
**When to use:** Any service that modifies persisted config.
**Example (from BluetoothManager):**
```cpp
// Constructor takes IConfigService*
BluetoothManager::BluetoothManager(IConfigService* configService, QObject* parent)
    : QObject(parent), configService_(configService) {}

// Usage: config write + save
configService_->setValue("connection.bt_name", alias);
configService_->save();
```

For ThemeService, a setter pattern is more appropriate (matches HostContext's existing `setConfigService()` style):
```cpp
// In ThemeService.hpp:
void setConfigService(IConfigService* svc) { configService_ = svc; }

// In ThemeService.cpp setTheme():
bool ThemeService::setTheme(const QString& themeId) {
    // ... existing load logic ...
    if (configService_) {
        configService_->setValue("display.theme", themeId);
        configService_->save();
    }
    return true;
}
```

### Pattern 2: SettingsToggle with Computed State
**What:** SettingsToggle normally binds to a `configPath` for auto-persistence. Here the toggle state is *derived* from whether `wallpaper_override` is non-empty, not directly persisted as its own key.
**When to use:** When toggle represents a computed condition rather than a standalone boolean config value.
**Key insight:** Don't use `configPath` on this toggle. Instead, manually control `checked` state by watching `ConfigService.onConfigChanged("display.wallpaper_override", ...)` and computing: `checked = (value !== "" && value !== undefined)`. Toggle OFF calls `ThemeService.setWallpaperOverride("")` + `ConfigService.setValue("display.wallpaper_override", "")` + `ConfigService.save()`.

### Pattern 3: Conditional Row Visibility
**What:** Use QML `visible` property to show/hide settings rows, with `rowIndex` renumbered for visual consistency.
**When to use:** Gating UI elements behind a toggle.
**Example (existing pattern from delete theme row):**
```qml
SettingsRow {
    visible: someCondition
    // ...
}
```

### Anti-Patterns to Avoid
- **Don't add a separate `display.custom_wallpaper_enabled` config key.** The toggle state is derivable from `display.wallpaper_override` being non-empty. Adding a separate key creates sync bugs.
- **Don't use SettingsToggle's configPath for the wallpaper toggle.** It has no standalone config key -- it's a UI gate derived from existing config.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Config persistence | Custom YAML write in ThemeService | IConfigService.setValue() + save() | Already handles YAML serialization, change notifications |
| Toggle component | New toggle widget | SettingsToggle (minus configPath) | Existing component, themed, accessible |
| Row show/hide animation | Custom animation system | QML `visible` + `Behavior on height` if desired | Built-in QML property animation |

## Common Pitfalls

### Pitfall 1: setTheme() Called During Startup
**What goes wrong:** main.cpp already calls `themeService->setTheme(savedTheme)` at startup (line 202). If we add config persistence to `setTheme()`, the startup call will write the already-saved theme back to config -- a no-op write but triggers `configChanged` signal.
**Why it happens:** Persistence added unconditionally to setTheme().
**How to avoid:** This is actually fine. `ConfigService::setValue()` only emits `configChanged` if the value actually changes (check YamlConfig::setValueByPath). But verify: if `setValueByPath` always returns true even for same-value writes, we'll get a spurious signal on startup. Check the implementation.
**Warning signs:** Unnecessary config writes on every startup.

### Pitfall 2: Toggle OFF Must Clear Override AND Persist
**What goes wrong:** Toggle OFF clears the wallpaper override in ThemeService but doesn't persist the empty string to config. On restart, old override reappears.
**How to avoid:** Toggle OFF must do: `ThemeService.setWallpaperOverride("")` + `ConfigService.setValue("display.wallpaper_override", "")` + `ConfigService.save()`. The FullScreenPicker already does this via its `configPath` binding, but the toggle's OFF action bypasses the picker.

### Pitfall 3: SettingsToggle Checked State Initialization
**What goes wrong:** Custom wallpaper toggle doesn't reflect current state on component load.
**How to avoid:** In `Component.onCompleted`, read `ConfigService.value("display.wallpaper_override")` and set checked = (value is non-empty string).

### Pitfall 4: "None" vs Empty String Semantics
**What goes wrong:** Confusing "none" (intentional no-wallpaper override) with "" (theme default / no override).
**Key distinction:**
- `""` (empty) = no override, use theme's wallpaper = toggle OFF
- `"none"` = intentional override meaning "no wallpaper at all" = toggle ON
- `"file://..."` = custom image override = toggle ON
**How to avoid:** Toggle checked = `value !== ""` (where value is the wallpaper_override config). This correctly treats "none" as ON.

### Pitfall 5: deleteTheme() Also Calls setTheme("default") Internally
**What goes wrong:** When deleting the active theme, `deleteTheme()` calls `setTheme("default")` which will now persist "default" to config. This is actually correct behavior -- it's a feature, not a bug.
**No action needed:** Just be aware this now auto-persists.

## Code Examples

### ThemeService Config Persistence Addition
```cpp
// ThemeService.hpp - add member and setter
private:
    IConfigService* configService_ = nullptr;
public:
    void setConfigService(IConfigService* svc) { configService_ = svc; }

// ThemeService.cpp - persist in setTheme()
bool ThemeService::setTheme(const QString& themeId) {
    auto it = themeDirectories_.find(themeId);
    if (it == themeDirectories_.end())
        return false;
    if (!loadTheme(it.value()))
        return false;
    resolveWallpaper();
    // Persist theme selection
    if (configService_) {
        configService_->setValue("display.theme", themeId);
        configService_->save();
    }
    return true;
}
```

### main.cpp Wiring
```cpp
// After themeService creation, before setTheme() call:
themeService->setConfigService(configService.get());
```

### Wallpaper Toggle in ThemeSettings.qml
```qml
// Custom Wallpaper toggle (not using configPath -- state derived from override value)
SettingsRow { rowIndex: 1
    Item {
        Layout.fillWidth: true
        implicitHeight: UiMetrics.rowH

        RowLayout {
            anchors.fill: parent
            anchors.leftMargin: UiMetrics.marginRow
            anchors.rightMargin: UiMetrics.marginRow
            spacing: UiMetrics.gap

            Text {
                text: "Custom Wallpaper"
                font.pixelSize: UiMetrics.fontBody
                color: ThemeService.onSurface
                Layout.fillWidth: true
            }

            Switch {
                id: wpToggle
                onToggled: {
                    if (!checked) {
                        // Clear override -- revert to theme wallpaper
                        ThemeService.setWallpaperOverride("")
                        ConfigService.setValue("display.wallpaper_override", "")
                        ConfigService.save()
                    }
                }
            }
        }

        SettingsHoldArea {
            anchors.fill: parent
            onShortClicked: wpToggle.checked = !wpToggle.checked
        }

        Component.onCompleted: {
            var v = ConfigService.value("display.wallpaper_override")
            wpToggle.checked = (v !== undefined && v !== "" && v !== null)
        }

        Connections {
            target: ConfigService
            function onConfigChanged(path, value) {
                if (path === "display.wallpaper_override") {
                    wpToggle.checked = (value !== undefined && value !== "" && value !== null)
                }
            }
        }
    }
}

// Wallpaper picker -- only visible when toggle is ON
SettingsRow { rowIndex: 2; visible: wpToggle.checked
    FullScreenPicker {
        flat: true
        label: "Wallpaper"
        configPath: "display.wallpaper_override"
        options: ThemeService.availableWallpaperNames
        values: ThemeService.availableWallpapers
        onActivated: function(index) {
            ThemeService.setWallpaperOverride(ThemeService.availableWallpapers[index])
        }
        Component.onCompleted: ThemeService.refreshWallpapers()
    }
}
```

### Discretion Recommendations

1. **Toggle component:** Use a custom inline toggle (not SettingsToggle) since SettingsToggle's `configPath` auto-persistence doesn't apply here. The toggle state is derived, not stored. See code example above.

2. **Animation:** No animation needed. The `visible` property instantly shows/hides. On a car touchscreen, instant feedback is better than smooth animations.

3. **Remove "Theme Default" from picker:** YES, remove it. When the toggle is ON, the user is explicitly choosing an override. "Theme Default" in the picker is confusing when toggle OFF already means "use theme default." Keep "None" in the picker since it's a distinct override ("no wallpaper at all").

4. **Strip "None" from picker:** NO, keep it. "None" means "I want no wallpaper at all" which is different from "use theme's wallpaper." It's a valid override choice.
