# Phase 03.1: Get AA Theme Information Working From Stream - Context

**Gathered:** 2026-03-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Flip the UiConfigRequest (0x8011) protocol direction from HU->Phone to Phone->HU, parse incoming Material You theming tokens, feed them to ThemeService, and send the acknowledgment response (0x8012) back. The Connected Device theme receives live phone colors end-to-end: phone pushes palette, HU applies it, UI updates, colors persist. Key format migrated from underscores to hyphens to match wire token names.

</domain>

<decisions>
## Implementation Decisions

### Protocol Direction Fix
- UiConfigRequest (0x8011) is Phone->HU (not HU->Phone as currently implemented)
- Remove the outbound `sendUiConfigRequest()` from AndroidAutoOrchestrator entirely
- Add inbound handler in VideoChannelHandler to parse 0x8011 as incoming UiConfigRequest
- UpdateHuUiConfigResponse (0x8012) is HU->Phone — send ACCEPTED status after receiving tokens
- Always respond ACCEPTED (no validation/rejection logic)

### Light/Dark Token Mapping
- `primary_configs` (field 1) = light mode tokens -> ThemeService day colors
- `secondary_configs` (field 2) = dark mode tokens -> ThemeService night colors
- Both lists applied independently to their respective color maps (not same palette for both)

### Key Format Migration
- Switch all ThemeService keys and theme YAML files from underscores to hyphens (e.g., `on_surface` -> `on-surface`, `surface_container_low` -> `surface-container-low`)
- Matches wire token key format — no normalization needed on receive
- All 5 theme YAML files updated (default, AMOLED, Ocean, Ember, connected-device)
- QML references that use camelCase Q_PROPERTY names are unaffected (only internal key storage changes)

### Theme Activation
- Manual selection only — user must pick "Connected Device" in theme settings
- No auto-switch on AA connect/disconnect
- When Connected Device is active, incoming colors apply immediately (instant swap, no animation)
- When a different theme is active, incoming colors are still parsed and cached to connected-device theme YAML (always cache)
- Switching to Connected Device mid-session applies cached colors immediately from YAML

### Live Color Behavior
- Handle multiple pushes per session — always apply the latest tokens
- Instant color swap via existing `colorsChanged` signal (QML bindings react automatically)
- Info-level log on receive: token count, whether applied or cached

### Submodule Update
- Update prodigy-oaa-protocol submodule pointer to latest commit (direction fix + token documentation)

### Test Updates
- Update existing `applyAATokens` tests for hyphenated keys
- Add test for parsing incoming UiConfigRequest message
- All tests must pass (currently 64)

### Claude's Discretion
- Signal/callback plumbing from VideoChannelHandler to ThemeService (via AndroidAutoOrchestrator or direct)
- Whether to separate light/dark token application into distinct method calls or a single call with mode parameter
- How to handle partial token sets (phone sends fewer than 16 tokens)
- Exact info log message format

</decisions>

<specifics>
## Specific Ideas

- Proto direction was confirmed via APK deep trace: GMS CarService UiStyle callback -> jrt token enum -> proto conversion
- The 16 wire tokens match what we already support: primary, on-surface, surface, surface-variant, surface-container-low, inverse-surface, inverse-on-surface, outline, outline-variant, background, text-primary, text-secondary, red, on-red, yellow, on-yellow
- Phase 3 already built the `applyAATokens()` method with persistence and test coverage — this phase wires it to the actual protocol
- The existing `persistConnectedDeviceTheme()` handles YAML caching — the "always cache" behavior means calling persist even when theme isn't active

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ThemeService::applyAATokens()`: Already accepts `QMap<QString, uint32_t>`, updates colors, persists, emits signal — needs minor tweak for always-cache behavior
- `ThemeService::persistConnectedDeviceTheme()`: Writes current colors to connected-device YAML — already works
- `VideoChannelHandler`: Already has 0x8012 case (currently parsing as inbound) — needs direction flip
- `AndroidAutoOrchestrator::sendUiConfigRequest()`: Currently sends 0x8011 outbound — to be removed
- 4 `applyAATokens` unit tests in test_theme_service.cpp — need key format update

### Established Patterns
- Protocol messages parsed in channel handlers, signals emitted up to orchestrator/service layer
- ThemeService colors accessed via Q_PROPERTY in QML (camelCase: `textPrimary`, `onSurface`)
- YAML keys use snake_case internally — migrating to hyphenated to match wire format
- `colorsChanged` signal triggers all QML color rebinding

### Integration Points
- `VideoChannelHandler` — add inbound 0x8011 handler, flip 0x8012 to outbound
- `AndroidAutoOrchestrator` — remove `sendUiConfigRequest()`, add signal handler for incoming tokens
- `ThemeService` — modify `applyAATokens()` to always cache (even when not active theme), update key format
- `config/themes/*.yaml` — all 5 themes: underscore keys -> hyphen keys
- `test_theme_service.cpp` — update key format in tests, add protocol parsing test

</code_context>
