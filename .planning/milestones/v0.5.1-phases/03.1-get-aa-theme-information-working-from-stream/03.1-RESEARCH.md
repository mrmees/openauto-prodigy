# Phase 03.1: Get AA Theme Information Working From Stream - Research

**Researched:** 2026-03-08
**Domain:** AA protocol message handling, ThemeService integration, protobuf parsing
**Confidence:** HIGH

## Summary

This phase flips the direction of two protocol messages (0x8011 UiConfigRequest and 0x8012 UpdateHuUiConfigResponse) and wires incoming phone theme tokens through to ThemeService. The codebase is well-structured for this -- all the pieces exist, they just need reconnecting in the correct direction. The VideoChannelHandler already handles inbound messages via `onMessage()` and sends outbound via `emit sendRequested()`. ThemeService already has `applyAATokens()` with persistence and tests.

The key format migration (underscore to hyphen) affects 5 theme YAML files, the `knownAATokenKeys` set, the `activeColor()` lookup keys in all Q_PROPERTY accessors, the `legacyKeyMap` migration targets, and the test data. QML is unaffected because Q_PROPERTY names are camelCase (`surfaceContainerLow`) and don't change.

**Primary recommendation:** Implement as a single plan with 3 logical steps: (1) key format migration across ThemeService + YAMLs + tests, (2) protocol direction flip in VideoChannelHandler + orchestrator, (3) always-cache behavior in applyAATokens.

## Architecture Patterns

### Established Message Flow Pattern
```
Phone -> TCP -> Messenger -> AASession::onMessage -> VideoChannelHandler::onMessage()
  -> [parse protobuf] -> emit signal with parsed data
  -> AndroidAutoOrchestrator [connected slot] -> ThemeService::applyAATokens()
```

This is the same pattern used for video focus, audio focus, navigation events, phone status, etc. Do NOT invent a new pattern.

### Established Outbound Response Pattern
```
[handler method] -> serialize protobuf -> emit sendRequested(channelId, messageId, data)
  -> AASession [connected in registerChannel] -> Messenger::sendMessage()
```

Used by: `handleSetupRequest()` (sends SETUP_RESPONSE), `handleVideoFocusRequest()` (sends VIDEO_FOCUS_INDICATION), `sendAck()` (sends ACK_INDICATION). The 0x8012 response follows this exact pattern.

### Pattern: Signal from VideoChannelHandler to Orchestrator

**Recommendation:** Add a new signal on VideoChannelHandler for incoming UI config tokens:

```cpp
// In VideoChannelHandler.hpp signals:
void uiConfigTokensReceived(const QMap<QString, uint32_t>& dayTokens,
                             const QMap<QString, uint32_t>& nightTokens);
```

This follows the existing pattern where VideoChannelHandler parses protocol-specific messages and emits domain-level signals (cf. `videoFocusChanged`, `streamStarted`). The orchestrator connects to this signal and forwards to ThemeService.

**Why separate day/night maps:** The proto has `primary_configs` (light) and `secondary_configs` (dark) as separate repeated fields. Parsing them into separate maps at the handler level keeps the boundary clean -- ThemeService doesn't need to know about proto structure.

### Pattern: Token Application with Always-Cache

**Recommendation:** Modify `applyAATokens()` to accept separate day/night maps and always cache:

```cpp
void ThemeService::applyAATokens(const QMap<QString, uint32_t>& dayTokens,
                                  const QMap<QString, uint32_t>& nightTokens)
{
    // Always update connected-device YAML (cache for later activation)
    // Only update live colors + emit colorsChanged if connected-device is active
}
```

This is simpler than a mode parameter and directly matches the proto structure.

### Anti-Patterns to Avoid
- **Direct ThemeService pointer in VideoChannelHandler:** The protocol library should not depend on app-level services. Use Qt signals to decouple.
- **Parsing proto in the orchestrator:** The handler layer is where proto parsing belongs. Orchestrator receives domain objects (maps of tokens).
- **Skipping the response:** The phone logs "Theming tokens accepted/rejected/Error" based on 0x8012. Always send ACCEPTED to confirm receipt.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Protobuf parsing | Manual byte parsing | `UiConfigRequest::ParseFromArray()` | Proto has oneof, repeated fields -- hand-parsing is error-prone |
| Color conversion | Custom ARGB unpacking | `QColor::fromRgba(uint32_t)` | Already used in existing `applyAATokens()` |
| YAML persistence | Manual file writing | Existing `persistConnectedDeviceTheme()` | Already handles theme dir resolution, YAML emit |
| Key normalization | Runtime underscore-to-hyphen conversion | One-time migration of all files and code | Migration is cleaner than runtime normalization |

## Common Pitfalls

### Pitfall 1: MessageId Constants Are Misnamed
**What goes wrong:** The constants in `MessageIds.hpp` have names that don't match the actual proto direction. `OVERLAY_SESSION_UPDATE` (0x8011) is actually `UiConfigRequest` (Phone->HU), and `UPDATE_HU_UI_CONFIG_REQUEST` (0x8012) is actually the `UpdateHuUiConfigResponse` message ID despite its name containing "REQUEST".
**Why it happens:** These were named before the direction was fully understood.
**How to avoid:** Use the hex values (0x8011, 0x8012) in comments alongside the constant names. The constant names in MessageIds.hpp are in the read-only submodule so they cannot be renamed here.
**Warning signs:** Confusing yourself about which is inbound vs outbound.

### Pitfall 2: Key Format Mismatch Between activeColor() and YAML
**What goes wrong:** After migrating YAML keys to hyphens, the `activeColor()` lookups in Q_PROPERTY accessors still use underscores (e.g., `activeColor("on_surface")`). Colors return transparent.
**Why it happens:** The migration has to touch both the YAML files AND all the C++ string literals used as lookup keys.
**How to avoid:** Grep for ALL underscore-style key strings in ThemeService.cpp and .hpp. There are ~30+ occurrences across: the 16 Q_PROPERTY accessor methods, `knownAATokenKeys` set, `legacyKeyMap` values, `outlineVariant()`, `pressed()`. Change them all.
**Warning signs:** All colors returning transparent after migration. Tests failing with color comparison mismatches.

### Pitfall 3: Forgetting the legacyKeyMap Migration Targets
**What goes wrong:** The `legacyKeyMap` in ThemeService.cpp maps old v1 key names to the current key names. Those target values (right side of the map) also need to change from underscore to hyphen.
**Why it happens:** The map is easy to overlook since it's for backwards compat with old theme files.
**How to avoid:** Update both sides of `legacyKeyMap` -- the values must match the new hyphenated key format.

### Pitfall 4: Test Data Theme YAML Keys
**What goes wrong:** Tests pass locally because test data still uses underscore keys, but the production code now expects hyphen keys.
**Why it happens:** Test data at `tests/data/themes/default/theme.yaml` is a separate copy from `config/themes/default/theme.yaml`.
**How to avoid:** Migrate BOTH the test data theme YAML and the config theme YAMLs simultaneously.

### Pitfall 5: Protobuf oneof Access Pattern
**What goes wrong:** Accessing `UiConfigEntry.config_value().value()` without checking `has_config_value()` first, or misunderstanding the oneof wrapper.
**Why it happens:** The proto uses `oneof value { UiConfigValue config_value = 2; }` which adds a layer of indirection.
**How to avoid:** Check `entry.has_config_value()` before accessing `entry.config_value().value()`. The `value()` on `UiConfigValue` returns the uint32 ARGB color.

### Pitfall 6: Submodule is Read-Only
**What goes wrong:** Attempting to fix the `OVERLAY_SESSION_UPDATE` / `UPDATE_HU_UI_CONFIG_REQUEST` constant names in the submodule.
**Why it happens:** The MessageIds.hpp constants have confusing names.
**How to avoid:** Do NOT modify anything under `libs/prodigy-oaa-protocol/`. Use the existing constant names with clarifying comments. The submodule update (pointer bump) must reference an existing commit in the upstream repo.

### Pitfall 7: VideoChannelHandler Switch Case Ordering
**What goes wrong:** The current switch in `onMessage()` handles 0x8012 (`UPDATE_HU_UI_CONFIG_REQUEST`) with a parse block, and lists 0x8011 (`OVERLAY_SESSION_UPDATE`) in the "newer AV message" fallthrough group. These need to be swapped.
**Why it happens:** The direction was previously wrong.
**How to avoid:** Move 0x8011 into a dedicated case block with UiConfigRequest parsing. Move 0x8012 to use it for sending (remove from switch or change to a log-only fallthrough). Actually -- 0x8012 is now outbound only, so it should never arrive inbound. Remove it from the switch or leave it in the debug fallthrough group.

## Code Examples

### Parsing Incoming UiConfigRequest (0x8011) in VideoChannelHandler

```cpp
// In onMessage() switch, replace the OVERLAY_SESSION_UPDATE fallthrough:
case oaa::AVMessageId::OVERLAY_SESSION_UPDATE: {
    // 0x8011: UiConfigRequest (Phone->HU) — Material You theming tokens
    oaa::proto::messages::UiConfigRequest request;
    if (!request.ParseFromArray(data.constData(), data.size())) {
        qWarning() << "[VideoChannel] failed to parse UiConfigRequest";
        break;
    }

    QMap<QString, uint32_t> dayTokens, nightTokens;
    if (request.has_config()) {
        const auto& config = request.config();
        for (int i = 0; i < config.primary_configs_size(); ++i) {
            const auto& entry = config.primary_configs(i);
            if (entry.has_config_value()) {
                dayTokens[QString::fromStdString(entry.key())] = entry.config_value().value();
            }
        }
        for (int i = 0; i < config.secondary_configs_size(); ++i) {
            const auto& entry = config.secondary_configs(i);
            if (entry.has_config_value()) {
                nightTokens[QString::fromStdString(entry.key())] = entry.config_value().value();
            }
        }
    }

    qInfo() << "[VideoChannel] UiConfigRequest received:"
            << dayTokens.size() << "light tokens,"
            << nightTokens.size() << "dark tokens";

    emit uiConfigTokensReceived(dayTokens, nightTokens);

    // Send 0x8012 ACCEPTED response
    oaa::proto::messages::UpdateHuUiConfigResponse resp;
    resp.set_status(oaa::proto::messages::THEMING_TOKENS_ACCEPTED);
    QByteArray respData(resp.ByteSizeLong(), '\0');
    resp.SerializeToArray(respData.data(), respData.size());
    emit sendRequested(channelId(), oaa::AVMessageId::UPDATE_HU_UI_CONFIG_REQUEST, respData);
    break;
}
```

**Note:** The response uses `UPDATE_HU_UI_CONFIG_REQUEST` (0x8012) as the message ID constant -- yes, the name is confusing but that IS the correct constant for the response message per the submodule's MessageIds.hpp.

### Connecting in AndroidAutoOrchestrator

```cpp
// In onNewConnection(), after wiring other videoHandler_ signals:
connect(&videoHandler_, &oaa::hu::VideoChannelHandler::uiConfigTokensReceived,
        this, [this](const QMap<QString, uint32_t>& dayTokens,
                     const QMap<QString, uint32_t>& nightTokens) {
    if (!themeService_) return;

    auto* theme = qobject_cast<oap::ThemeService*>(
        dynamic_cast<QObject*>(themeService_));
    if (theme) {
        theme->applyAATokens(dayTokens, nightTokens);
    }
});
```

**Note:** The orchestrator has `themeService_` as `IThemeService*`. The `applyAATokens` method is on `ThemeService` (concrete class), not the interface. Either: (a) add `applyAATokens` to `IThemeService`, (b) use `dynamic_cast`/`qobject_cast` as shown, or (c) store a `ThemeService*` directly. Option (a) is cleanest.

### Modified applyAATokens with Always-Cache and Separate Day/Night

```cpp
void ThemeService::applyAATokens(const QMap<QString, uint32_t>& dayArgbTokens,
                                  const QMap<QString, uint32_t>& nightArgbTokens)
{
    // Build color maps from ARGB tokens
    QMap<QString, QColor> newDayColors, newNightColors;
    for (auto it = dayArgbTokens.constBegin(); it != dayArgbTokens.constEnd(); ++it) {
        if (knownAATokenKeys.contains(it.key()))
            newDayColors[it.key()] = QColor::fromRgba(it.value());
    }
    for (auto it = nightArgbTokens.constBegin(); it != nightArgbTokens.constEnd(); ++it) {
        if (knownAATokenKeys.contains(it.key()))
            newNightColors[it.key()] = QColor::fromRgba(it.value());
    }

    // Always cache to connected-device YAML (even if not active theme)
    // ... update cached day/night colors, call persistConnectedDeviceTheme()

    if (themeId_ == "connected-device") {
        // Apply live
        for (auto it = newDayColors.constBegin(); it != newDayColors.constEnd(); ++it)
            dayColors_[it.key()] = it.value();
        for (auto it = newNightColors.constBegin(); it != newNightColors.constEnd(); ++it)
            nightColors_[it.key()] = it.value();
        emit colorsChanged();
        qInfo() << "[Theme] Applied" << newDayColors.size() << "day +"
                << newNightColors.size() << "night AA tokens (live)";
    } else {
        qInfo() << "[Theme] Cached" << newDayColors.size() << "day +"
                << newNightColors.size() << "night AA tokens (theme not active)";
    }
}
```

## Open Questions

1. **IThemeService extension vs cast**
   - What we know: `applyAATokens` is on ThemeService but not IThemeService. Orchestrator holds `IThemeService*`.
   - What's unclear: Whether to add to interface or cast.
   - Recommendation: Add `applyAATokens(const QMap<QString, uint32_t>&, const QMap<QString, uint32_t>&)` to `IThemeService` as a virtual method with empty default implementation. Cleanest approach, no casts needed.

2. **Partial token sets**
   - What we know: Phone typically sends all 16 tokens, but proto allows fewer.
   - What's unclear: Whether phone ever sends partial sets in practice.
   - Recommendation: Accept whatever tokens arrive. The `knownAATokenKeys` filter already handles unknown keys. Partial sets just update fewer colors. Existing fallback colors in the connected-device YAML handle the rest.
