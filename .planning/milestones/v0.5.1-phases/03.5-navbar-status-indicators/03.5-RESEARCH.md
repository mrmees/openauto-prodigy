# Phase 03.5: Navbar Status Indicators - Research

**Researched:** 2026-03-09
**Domain:** AA protocol data extraction + Qt/QML UI indicators
**Confidence:** HIGH

## Summary

This phase adds battery and signal strength indicators to the navbar by extracting data from the AA protocol and exposing it to QML, plus telling the phone to hide its own status bar elements. The work touches three layers: (1) protocol handling in the open-android-auto submodule (new battery handler, signal field extraction), (2) data bridging via AndroidAutoOrchestrator properties, and (3) QML layout changes in Navbar/NavbarControl.

The protocol pieces are gold-confidence -- both `BatteryStatusNotification` (control channel msg 23) and `PhoneStatusUpdate.signal_strength` (field 2) are APK-verified protos. The `hidden_ui_elements` field on `AdditionalVideoConfig` is also gold-confidence. The main risk is that re-adding `AdditionalVideoConfig` to SDP VideoConfig may interfere with margins (as `ui_theme` did previously) -- a fallback path via runtime `UpdateUiConfigRequest` (0x800B) exists but is deferred.

**Primary recommendation:** Add battery/signal Q_PROPERTYs to AndroidAutoOrchestrator (not a new service), wire protocol signals, and expand the NavbarControl center zone with flanking MaterialIcon indicators.

## Architecture Patterns

### Data Flow
```
Phone → AA Protocol → ControlChannel (battery) / PhoneStatusChannelHandler (signal)
     → AndroidAutoOrchestrator (Q_PROPERTY bridge)
     → QML Navbar (binding)
```

### Pattern: Q_PROPERTY on Orchestrator (Recommended)

Add battery/signal properties directly to `AndroidAutoOrchestrator` rather than creating a new service. Rationale:
- Orchestrator already owns the protocol handlers (`phoneStatusHandler_`, control channel access via `session_->controlChannel()`)
- Orchestrator already bridges protocol events to the event bus (media, nav, phone call)
- CompanionListenerService has a separate battery source (companion app) -- these are different data paths
- A new service would need to be wired through IHostContext, adding complexity for just 4 properties

**New Q_PROPERTYs on AndroidAutoOrchestrator:**
```cpp
Q_PROPERTY(int phoneBatteryLevel READ phoneBatteryLevel NOTIFY phoneBatteryChanged)
Q_PROPERTY(bool phoneCharging READ isPhoneCharging NOTIFY phoneBatteryChanged)
Q_PROPERTY(int phoneSignalStrength READ phoneSignalStrength NOTIFY phoneSignalChanged)
Q_PROPERTY(bool aaConnected READ isAaConnected NOTIFY connectionStateChanged)
```

The `aaConnected` convenience property derives from existing `connectionState` (true when Connected or Backgrounded).

### Pattern: Reset on Disconnect

In `teardownSession()`, reset battery/signal to defaults (-1 or 0) before emitting signals. This ensures QML indicators disappear immediately on disconnect without stale data.

### Anti-Patterns to Avoid
- **Don't create a PhoneTelemetryService** -- overkill for 4 read-only properties that only exist during AA sessions
- **Don't poll for battery/signal** -- the phone pushes these values; just handle the incoming messages
- **Don't persist battery/signal to YAML** -- these are ephemeral session data

## Protocol Details

### Battery Status (Control Channel, msg ID 23 / 0x17)

```protobuf
// confidence: gold [apk_deep_trace]
message BatteryStatusNotification {
    optional uint32 battery_level = 1;  // percentage (0-100)
    optional uint32 time_remaining_s = 2;  // discharge time in seconds
    optional bool critical_battery = 3;
}
```

**Current state:** ControlChannel.cpp does NOT handle msg 23 -- it falls through to `unknownMessage` signal. The proto exists and is compiled.

**Required change (SUBMODULE):** Add `case 0x17:` to `ControlChannel::onMessage()` that parses `BatteryStatusNotification` and emits a new signal. This requires a change to the `open-android-auto` submodule.

**CRITICAL:** The submodule is read-only per project rules. The orchestrator currently connects to `session_->controlChannel()` signals. Two options:
1. **Modify the submodule** (preferred) -- add battery signal to ControlChannel
2. **Use `unknownMessage` signal** (workaround) -- connect to `unknownMessage(uint16_t, QByteArray)`, check for msg ID 0x17, parse in orchestrator

Recommend option 2 as a workaround to avoid submodule changes, with a note to upstream the change later. The `unknownMessage` signal already exists and carries the full payload.

### Signal Strength (PhoneStatus Channel, msg 0x8001)

```protobuf
message PhoneStatusUpdate {
    repeated PhoneCall calls = 1;
    optional uint32 signal_strength = 2;  // confidence: gold [deep_trace]
}
```

**Current state:** `PhoneStatusChannelHandler::handlePhoneStatus()` parses the message but ignores the `signal_strength` field entirely. It only processes `calls`.

**Required change (SUBMODULE):** Add `signal_strength` extraction and a new signal emission. Again, submodule is read-only.

**Workaround:** Same `unknownMessage` approach won't work here since the message IS handled. Options:
1. **Modify submodule** -- add `signalStrengthChanged(uint32_t)` signal to PhoneStatusChannelHandler
2. **Parse raw on control/phone channel** -- not feasible since the handler consumes the message

Recommend modifying the submodule for this one -- it's a 3-line addition to an existing handler. Note it for upstream.

### Signal Strength Value Range

**Confidence: LOW** -- The proto docs note: "The exact scale is not documented in the proto -- Android typically uses 0-4 bars, but the raw value may be a different range. Test with your target phone to calibrate."

Android's `ServiceState.getLevel()` returns 0-4 (NONE/POOR/MODERATE/GOOD/GREAT). The AA APK class `hll.java` converts `CarPhoneStatus` to proto, which likely passes the Android signal level directly.

**Recommended mapping (conservative, 0-4 range assumption):**
| Value | Icon | Codepoint |
|-------|------|-----------|
| 0 | signal_cellular_0_bar | \uf0a8 |
| 1 | signal_cellular_1_bar | \uf0a9 |
| 2 | signal_cellular_2_bar | \uf0aa |
| 3 | signal_cellular_3_bar | \uf0ab |
| 4 | signal_cellular_4_bar | \uf0ac |

If the value exceeds 4, clamp to 4. If absent (not sent), hide the indicator entirely.

|---------|-------------|-------------|-----|
| Icon rendering | Custom drawing code | MaterialIcon QML component | Already handles DPI, color, sizing |

## Common Pitfalls

### Pitfall 1: AdditionalVideoConfig Breaking Margins
**What goes wrong:** Setting ANY field on AdditionalVideoConfig (like ui_theme) caused the phone to ignore margin_width/margin_height on VideoConfig.
**Why it happens:** Phone-side behavior -- certain AVC fields trigger different rendering paths.
**How to avoid:** Set ONLY `hidden_ui_elements` -- no `ui_theme`, no resolution ranges. Test margins after enabling. Have a revert path ready.
**Warning signs:** AA video content suddenly fills the entire screen, ignoring navbar space.

### Pitfall 2: Stale Data After Disconnect
**What goes wrong:** Battery shows "87%" after phone disconnects because properties weren't reset.
**Why it happens:** Q_PROPERTYs retain last value until explicitly changed.
**How to avoid:** Reset to sentinel values (-1) in `teardownSession()`. QML binds visibility to `aaConnected` AND valid value.

### Pitfall 3: Submodule Modification Scope Creep
**What goes wrong:** Making changes to open-android-auto submodule without tracking them.
**Why it happens:** Submodule is read-only per project rules but we need new signals.
**How to avoid:** Make minimal changes, document them, push to the submodule repo separately.

### Pitfall 4: Signal Strength Value Assumptions
**What goes wrong:** Assuming 0-4 range but phone sends 0-5 or 0-100.
**Why it happens:** Proto docs don't specify the range.
**How to avoid:** Implement with configurable thresholds. Log raw values on first connection. Clamp to valid range.

### Pitfall 5: Material Icon Codepoint Mismatches
**What goes wrong:** Icon shows as blank square because codepoint doesn't exist in bundled font.
**Why it happens:** Material Icons has multiple font variants (Outlined, Filled, Rounded) with different coverage.
**How to avoid:** Test all icon codepoints against the bundled font before finalizing. Use only icons verified to render.

### Battery via unknownMessage workaround (AndroidAutoOrchestrator)
```cpp
// In onNewConnection(), after session creation:
connect(session_->controlChannel(), &oaa::ControlChannel::unknownMessage,
        this, [this](uint16_t msgId, const QByteArray& payload) {
    if (msgId == 0x0017) {  // BatteryStatusNotification
        oaa::proto::messages::BatteryStatusNotification batt;
        if (batt.ParseFromArray(payload.constData(), payload.size())) {
            phoneBatteryLevel_ = batt.has_battery_level() ? static_cast<int>(batt.battery_level()) : -1;
            phoneCharging_ = batt.has_critical_battery() ? false : false; // charging not in proto
            emit phoneBatteryChanged();
        }
    }
});
```

**WAIT** -- the `BatteryStatusNotification` proto does NOT have a charging field. It has `battery_level`, `time_remaining_s`, and `critical_battery`. Charging state is NOT available from this protocol message.

**Resolution:** Check if Android sends charging info elsewhere, or infer from rising battery_level over time. For now, only show battery level without charging indicator. The CONTEXT.md mentions charging icons, but the protocol may not support it. Flag for testing.

## Charging State Investigation

**Finding:** The `BatteryStatusNotification` proto has NO charging field. Fields are:
- `battery_level` (uint32, percentage)
- `time_remaining_s` (uint32, discharge prediction)
- `critical_battery` (bool)

**Confidence: HIGH** (gold proto, APK-traced)

The proto comment says "Battery level from BatteryManager.EXTRA_LEVEL" and "Time from getBatteryDischargePrediction()". The APK sender class (`hna.java`) gets data from `BatteryManager`.

Android's BatteryManager also provides `BATTERY_STATUS_CHARGING` but it's NOT included in this proto. Possible approaches:
1. **Infer from time_remaining_s = 0 or absent** -- if not discharging, might be charging (SPECULATIVE)
2. **Skip charging indicators** -- just show battery level, no charging variant icons
3. **Check if charging data comes on another channel** -- unlikely given gold-level proto analysis

**Recommendation:** Skip charging icons for now. Show battery fill level only. If users request charging indicator, investigate protocol captures later.

   - What's unclear: Actual values sent by phones in practice
   - Recommendation: Log raw values on first connection, implement with 0-4 assumption and clamp

2. **Charging state availability**
   - What we know: `BatteryStatusNotification` has no charging field
   - What's unclear: Whether charging info comes through another mechanism
   - Recommendation: Skip charging icons, show battery level only

3. **Material Icon codepoint coverage**
   - What we know: The bundled font has volume/brightness icons working
   - What's unclear: Which battery/signal codepoints are in the bundled font
   - Recommendation: Test during implementation, have fallback icon set

4. **AdditionalVideoConfig margin interaction**
   - What we know: `ui_theme` field broke margins; `hidden_ui_elements` is a different field
   - What's unclear: Whether `hidden_ui_elements` alone also affects margins
   - Recommendation: Test immediately after implementation; have a revert path
