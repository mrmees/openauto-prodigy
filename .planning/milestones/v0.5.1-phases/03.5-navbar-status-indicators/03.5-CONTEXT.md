# Phase 03.5: Navbar Status Indicators - Context

**Gathered:** 2026-03-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Add battery and signal strength indicators to the navbar using AA protocol data, and tell the phone to hide its own status bar elements (clock, battery, signal) from the projected AA UI via `hidden_ui_elements` in `AdditionalVideoConfig`. Indicators are display-only — no new touch zones.

</domain>

<decisions>
## Implementation Decisions

### Data Sources
- AA protocol only — no companion app data for indicators
- Parse `BatteryStatusNotification.battery_level` from control channel (new handler code)
- Parse `PhoneStatusUpdate.signal_strength` from phone status channel (field exists but not parsed yet)
- Both protos are gold-confidence in the submodule

### Visibility & Lifecycle
- Indicators visible whenever an AA session is connected (not just when AA is the active plugin view)
- Disappear when AA disconnects — no stale data persistence
- Visual only — no touch interaction, no new evdev zones needed

### Navbar Layout
- Battery and signal icons flank the clock in the center zone
- Reduce outer control zones from 25% to 15% (volume and brightness closer to edges)
- Center zone expands from 50% to 70% — clock remains centered with indicators on either side
- Layout works for all 4 edge positions (horizontal and vertical)

### AA Status Bar Hiding
- Re-add `AdditionalVideoConfig` to SDP VideoConfig, setting ONLY `hidden_ui_elements` (no `ui_theme` — that broke margins)
- Hide all three: `UI_ELEMENT_CLOCK`, `UI_ELEMENT_BATTERY_LEVEL`, `UI_ELEMENT_PHONE_SIGNAL`
- Conditional on `navbar.show_during_aa` config — if navbar is hidden during AA, don't hide phone status bar elements
- Check happens at SDP build time only (session start); changing navbar visibility requires AA reconnect
- If `hidden_ui_elements` re-breaks margins (like `ui_theme` did), fall back to runtime `UpdateUiConfigRequest` (0x800B) after session establishes

### Visual Design
- Battery: Material fill-level icons (battery_full → battery_6_bar → battery_4_bar → battery_3_bar → battery_1_bar → battery_alert)
- Charging: battery_charging_* variant icons
- Signal: Material bar-level icons (signal_cellular_4_bar through signal_cellular_0_bar)
- Icon size: `UiMetrics.iconSize` (same as volume/brightness icons)
- Color: monochrome (white on black during AA, themed otherwise), except battery turns `ThemeService.error` (red) below 20% — during AA use hardcoded red

### Claude's Discretion
- Exact icon-to-percentage thresholds for battery fill levels
- Signal strength value-to-bars mapping (protocol value range needs research)
- Spacing between indicators and clock
- Vertical layout stacking order for indicators
- How to structure the new battery/signal properties (new service vs properties on orchestrator)

</decisions>

<specifics>
## Specific Ideas

- "We need to make sure we're only disabling the AA display items when the navbar is showing. Otherwise the AA framework needs to display these items."
- Outer controls (volume/brightness) should be closer to screen edges — current 25% zones are too generous

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `NavbarControl.qml`: Individual control component — center control shows clock, could add indicator items alongside
- `MaterialIcon` QML component: Already used for volume/brightness icons in navbar
- `PhoneStatusChannelHandler`: Already handles call state, needs `signal_strength` field extraction added
- `CompanionListenerService`: Has battery Q_PROPERTYs pattern to follow (phoneBattery, phoneCharging, batteryChanged signal)
- `UiMetrics.iconSize`: Existing DPI-scaled icon size property

### Established Patterns
- Navbar uses `aaActive` bool to switch between themed colors and hardcoded black/white during AA
- `AndroidAutoOrchestrator` bridges protocol signals to EventBus (aa.phone.call, aa.media.state pattern)
- Q_PROPERTY + NOTIFY signal pattern for exposing protocol data to QML (see CompanionListenerService)

### Integration Points
- `ServiceDiscoveryBuilder.cpp:237` — `AdditionalVideoConfig` was removed; needs conditional re-addition with only `hidden_ui_elements`
- `AndroidAutoOrchestrator` — needs to connect battery/signal signals from handlers and expose to QML
- `Navbar.qml` — center zone (hControl1/vControl1) needs battery + signal icons added
- `NavbarControl.qml` — center control needs layout expansion for flanking indicators
- `AdditionalVideoConfigData.proto:22` — `hidden_ui_elements` field (repeated UIElement enum)
- `BatteryStatusMessage.proto` — `BatteryStatusNotification` message (battery_level, time_remaining_s, critical_battery)
- `PhoneStatusMessage.proto:19` — `signal_strength` field on PhoneStatusUpdate

</code_context>
