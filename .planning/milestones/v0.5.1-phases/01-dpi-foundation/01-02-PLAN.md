<task type="auto">
  <name>Task 1: Replace UiMetrics resolution-ratio with DPI-based scaling</name>
  <files>qml/controls/UiMetrics.qml</files>
  <action>
    Replace the scaleH/scaleV computation in UiMetrics.qml. The current lines:
    ```
    readonly property real scaleH: _winW / 1024.0
    readonly property real scaleV: _winH / 600.0
    ```

    Replace with DPI-based computation. Add these properties BEFORE scaleH/scaleV:
    ```javascript
    // --- DPI-based scaling (replaces hardcoded 1024/600 resolution ratio) ---
    readonly property real _screenSize: DisplayInfo ? DisplayInfo.screenSizeInches : 7.0
    readonly property real _diagPx: Math.sqrt(_winW * _winW + _winH * _winH)
    readonly property real _dpi: _diagPx / _screenSize
    readonly property real _referenceDpi: 160.0  // Android mdpi standard
    ```

    Then replace scaleH/scaleV with:
    ```javascript
    // DPI-based: actual_DPI / 160 (uniform for square pixels, preserves dual-axis infrastructure)
    readonly property real scaleH: _dpi / _referenceDpi
    readonly property real scaleV: _dpi / _referenceDpi
    ```

    Update the Component.onCompleted log to include DPI info:
    ```javascript
    console.log("[UiMetrics] window:", _winW + "x" + _winH,
                "screen:", _screenSize + "\"",
                "dpi:", Math.round(_dpi),
                "scaleH:", scaleH.toFixed(3), "scaleV:", scaleV.toFixed(3),
                ...rest unchanged...);
    ```

    Everything downstream (autoScale, _autoFontScale, scale, all tokens) is unchanged --
    they all derive from scaleH/scaleV which now come from DPI instead of resolution ratio.

    Key: Do NOT change any other properties. The entire token system (rowH, touchMin, fonts, etc.)
    flows from scaleH/scaleV through autoScale and _autoFontScale. Changing the source is sufficient.
  </action>

    <automated>cd /home/matt/claude/personal/openautopro/openauto-prodigy/build && cmake --build . -j$(nproc) && ctest --output-on-failure</automated>
  </verify>
  <done>UiMetrics.qml uses _dpi / 160.0 for scaleH/scaleV. No hardcoded 1024 or 600 in scale computation. All existing tests still pass (QML change, no C++ test impact). Startup log shows DPI info.</done>

    BEFORE the "General" SectionHeader (line 18). This should be the very first visible
    element in Display settings.

    Use the existing ReadOnlyField control with a computed value (not configPath, since
    this combines two DisplayInfo properties):

    ```qml
    ReadOnlyField {
        label: "Screen"
        value: {
            var size = DisplayInfo ? DisplayInfo.screenSizeInches : 7.0;
            var dpi = DisplayInfo ? DisplayInfo.computedDpi : 170;
            return size.toFixed(1) + "\" / " + dpi + " PPI";
        }
    }
    ```

    This shows something like: `Screen    7.0" / 170 PPI`

    Place it before the SectionHeader so it reads as a top-level info bar, not part of
    any settings section. It's informational only -- no interaction, no configPath binding.
