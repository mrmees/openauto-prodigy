# Phase 03.4: Fix Android Auto Rendering - Research

**Researched:** 2026-03-09
**Domain:** AA protocol video margins + QML rendering pipeline
**Confidence:** HIGH

## Summary

The phone (S25 Ultra) is ignoring `margin_width`/`margin_height` in the SDP VideoConfig, causing AA to render full-screen while the navbar overlaps the content. Investigation shows only two SDP changes since v0.5.0 (when margins worked): `color_scheme_support` on AVChannel field 9, and `AdditionalVideoConfig` on each VideoConfig. Both were ruled out by Matt through testing.

The code diff since v0.5.0 reveals no layout or rendering changes -- AndroidAutoMenu.qml only had color reference renames, Shell.qml is identical, and AndroidAutoPlugin.cpp only added a `setThemeService()` call. The proto submodule change (ed530470 to 69733b8a) did NOT modify VideoConfigData.proto -- `margin_width` (field 3) and `margin_height` (field 4) are unchanged.

The most likely root cause is a phone-side AA app update that changed how margins are processed. The fix should be rendering-side: make VideoOutput display correctly regardless of whether the phone honors margins.

**Primary recommendation:** Change VideoOutput fillMode from `PreserveAspectCrop` to `PreserveAspectFit` so the full video frame is visible (letterboxed) within the navbar-inset content area. This works whether or not the phone honors margins.

## Architecture Patterns

### Current Rendering Pipeline

```
Phone renders video at {resolution} (e.g., 1280x720)
  --> margin_width/margin_height SHOULD shrink content to centered sub-region
  --> Phone sends full-resolution frames (content + black bars)
  --> Shell.qml insets content area by navbar.barThick (UiMetrics.navbarThick)
  --> VideoOutput fills inset area with PreserveAspectCrop
  --> Result: if margins ignored, Crop cuts into actual content at top AND bottom
```

### Why PreserveAspectCrop Causes the Problem

With navbar visible (default), Shell.qml insets content area to 1024x544 (600 - 56). The 1280x720 video frame (16:9 = 1.778) is wider than the inset area (1024/544 = 1.882). With `PreserveAspectFit`, the video would letterbox with small black bars on the sides. With `PreserveAspectCrop`, the video fills height and crops width -- but 720p in 544px viewport means scale = 544/720 = 0.756, scaled width = 1280*0.756 = 967px < 1024px, so Qt would instead scale by width (1024/1280 = 0.8) making height = 720*0.8 = 576px, cropping 32px (16px top + 16px bottom). This explains the "top AND bottom cut off" symptom.

**If the phone were honoring margins:** The content sub-region would be smaller (e.g., 1280x680 for 40px margin_height). PreserveAspectCrop would crop the black bars, not the content. Without margins, Crop cuts into actual content.

### The Fix: PreserveAspectFit

Changing to `PreserveAspectFit`:
- Video scales to fit entirely within the content area (1024x544)
- Small letterbox bars appear (acceptable -- black on black background)
- ALL content visible regardless of whether phone honors margins
- If margins work: slightly smaller display but fully visible
- If margins don't work: letterbox bars compensate

### Alternative Fix: VideoMarginConfig in AdditionalVideoConfig

The proto `AdditionalVideoConfig` has a `margin_configs` field (field 7) containing `VideoMarginConfig` with `VideoInsets` (left/top/right/bottom). This is a more granular margin system discovered in APK decompilation. Cross-referencing with aa-proxy-rs shows their equivalent `UiConfig` (field 11) has `margins` as an `Insets` message at field 1 -- different field numbers but same concept.

The phone may prefer `margin_configs` over the legacy `margin_width`/`margin_height` when `additional_config` is present. However, since Matt tested removing `additional_config` entirely and margins still didn't work, this is not the primary issue. Worth investigating later for the "correct long-term fix."

### Touch Coordinate Implications

**CRITICAL:** If fillMode changes from Crop to Fit, the touch coordinate mapping in `EvdevTouchReader` must be updated to match. Currently `computeLetterbox()` calculates coordinates based on whether the video is cropped or fit. The fit-mode math is already implemented (used when no sidebar is present per the rendering doc).

With Fit mode:
- `visibleAAWidth = aaWidth` (full content width in AA coords)
- `visibleAAHeight = aaHeight` (full content height in AA coords)
- Touch maps linearly from video's on-screen position to full AA range
- Letterbox regions are dead zones (not forwarded to AA)

The `touch_screen_config` in SDP should remain set to content dimensions (after margin subtraction). If margins are ignored by the phone, the phone expects touches in full 1280x720 space, which means `touch_screen_config` should match video resolution when margins are not honored.

**Subtle issue:** If we set margins in SDP but the phone ignores them, our `touch_screen_config` says (e.g.) 1280x680 but the phone expects 1280x720. This would cause vertical touch misalignment. The rendering fix should verify touch accuracy and potentially adjust `touch_screen_config` if margins are confirmed broken.

### Most Likely: Phone-Side AA App Update

Samsung AA app receives periodic updates. Between v0.5.0 (working) and now, the phone's AA app likely updated and changed how it processes `margin_width`/`margin_height`. This is a known issue in the AA ecosystem -- head unit developers report margin behavior changing between phone updates.

### Recommendation: Skip Bisect, Go Straight to Rendering Fix

Bisecting would confirm the regression is code-side or phone-side, but:
1. Building and testing each commit requires Pi deployment + phone pairing (slow)
2. If it IS phone-side (most likely), bisect wastes time
3. The rendering fix is needed regardless (defensive coding)
4. The rendering fix is small (fillMode change + touch coord verification)

**Do the rendering fix first.** If someone later wants to investigate the protocol-level margin issue, the proto `margin_configs` in `AdditionalVideoConfig` is the next avenue.

## Common Pitfalls

### Pitfall 1: Changing fillMode Without Updating Touch Mapping
**What goes wrong:** Video displays correctly but touch coordinates are offset
**Why it happens:** EvdevTouchReader's letterbox calculation assumes Crop mode when navbar is present
**How to avoid:** After changing fillMode, verify touch accuracy on Pi. Update `computeLetterbox()` if needed.
**Warning signs:** Touches landing offset from visual target, especially at edges

### Pitfall 2: Breaking No-Navbar Mode
**What goes wrong:** AA rendering changes inadvertently affect fullscreen (no-navbar) mode
**Why it happens:** The fix path might hardcode assumptions about navbar presence
**How to avoid:** Test both modes: `navbar.show_during_aa = true` AND `false`

### Pitfall 3: Assuming Margins Are Permanently Broken
**What goes wrong:** Removing margin_width/margin_height from SDP, losing the "correct" fix
**Why it happens:** Phone ignores them NOW, but a future AA update might respect them again
**How to avoid:** Keep margins in SDP. The rendering fix should work WITH or WITHOUT margins.

### Pitfall 4: Debug Overlay Coordinates Broken After Fix
**What goes wrong:** Green crosshair debug overlay shows touches at wrong positions
**Why it happens:** The overlay in AndroidAutoMenu.qml uses contentW/contentH for coordinate mapping
**How to avoid:** Verify debug overlay still aligns after fillMode change

## Code Examples

### Current: PreserveAspectCrop (broken)
```qml
// qml/applications/android_auto/AndroidAutoMenu.qml
VideoOutput {
    id: videoOutput
    anchors.fill: parent
    visible: androidAutoMenu.projecting
    fillMode: VideoOutput.PreserveAspectCrop  // Crops content when margins ignored
}
```

### Fix: PreserveAspectFit
```qml
VideoOutput {
    id: videoOutput
    anchors.fill: parent
    visible: androidAutoMenu.projecting
    fillMode: VideoOutput.PreserveAspectFit  // Letterbox: all content visible
}
```

### aa-proxy-rs UiConfig Proto (Reference)
```protobuf
// aa-proxy-rs shows field 11 sub-message as UiConfig with margins at field 1
message UiConfig {
  optional Insets margins = 1;          // Our AdditionalVideoConfig has this at field 7
  optional Insets content_insets = 2;
  optional Insets stable_content_insets = 3;
  optional UiTheme ui_theme = 4;        // Same field number as ours
}
```

Note: aa-proxy-rs and our proto disagree on the internal field layout of the field-11 sub-message. Both have `ui_theme` at field 4, but margins are at field 1 in aa-proxy-rs vs field 7 (wrapped in VideoMarginConfig) in ours. This discrepancy is worth investigating for the long-term margin fix but is NOT the cause of the current regression (Matt tested removing `additional_config` entirely).
