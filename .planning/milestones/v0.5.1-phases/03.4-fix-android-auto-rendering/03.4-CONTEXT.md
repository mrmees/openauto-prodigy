# Phase 03.4: Fix Android Auto Rendering - Context

**Gathered:** 2026-03-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix AA video rendering so the navbar doesn't overlap the Android Auto content. The phone is ignoring our margin_width/margin_height values in the SDP VideoConfig, causing it to render full-screen while our navbar floats on top.

</domain>

<decisions>
## Implementation Decisions

### Problem Statement
- Phone (S25 Ultra) ignores `margin_width`/`margin_height` in VideoConfig SDP
- Tested at 480p (margins 0x55) and 720p (margins 0x40) — both ignored
- Fresh pair (forget car + re-pair) — still ignored
- Disabling `color_scheme_support` and `AdditionalVideoConfig` — still ignored
- Margins were working before the v0.5.1 milestone work began
- Top AND bottom of AA content are cut off (PreserveAspectCrop cropping both edges)

### Root Cause Investigation Needed
- Something changed between v0.5.0 (working margins) and current dev/0.5.1 that broke margin handling
- The `color_scheme_support` and `AdditionalVideoConfig` additions were ruled out as cause
- Need to bisect: find the exact commit where margins stopped working
- Check if proto submodule v1.0 changed VideoConfigData field numbering

### Approach
- If margins can't be fixed at the protocol level, fall back to rendering-side fix:
  - Change VideoOutput fillMode from PreserveAspectCrop to PreserveAspectFit (letterbox)
  - Or explicitly size/clip the VideoOutput to account for the navbar
- Margins are the correct long-term fix (phone renders smaller, no wasted pixels)

### Known Facts
- `ServiceDiscoveryBuilder::calcMargins()` correctly computes margins (verified via logs)
- `ServiceDiscoveryBuilder::calcNavbarViewport()` correctly reads `navbar.show_during_aa` from config
- C++ hardcodes navbar thickness as 56px; QML uses DPI-scaled `UiMetrics.navbarThick` (Math.round(56 * scale))
- Config default for `ui.tokens.navbarThick` is 0 in YamlConfig.cpp — should probably be non-zero or removed
- `wantsFullscreen()` correctly returns false, Shell.qml correctly insets content area
- Plugin views are parented to `pluginContentHost` (clip: true, inside margin-inset ColumnLayout)

### Claude's Discretion
- Whether to bisect commits or go straight to rendering-side fix
- Exact VideoOutput fillMode and sizing approach
- Whether to fix the navbarThick config default (0 → remove or set 56)

</decisions>

<code_context>
## Existing Code Insights

### Key Files
- `src/core/aa/ServiceDiscoveryBuilder.cpp` — builds SDP with margin_width/margin_height per VideoConfig
- `src/core/aa/ServiceDiscoveryBuilder.hpp` — `computeContentDimensions()` static method (single source of truth)
- `src/plugins/android_auto/AndroidAutoPlugin.cpp` — reads navbar config, sets display dims, wantsFullscreen()
- `qml/applications/android_auto/AndroidAutoMenu.qml` — VideoOutput with PreserveAspectCrop fillMode
- `qml/components/Shell.qml` — ColumnLayout with navbar margin insets, pluginContentHost (clip:true)
- `qml/controls/UiMetrics.qml:81` — navbarThick token (DPI-scaled)
- `src/core/YamlConfig.cpp:151` — navbarThick default = 0 (suspect)

### Margin Pipeline
1. `ServiceDiscoveryBuilder::calcMargins()` computes margins from display dims + navbar config
2. Margins set on each VideoConfig in SDP → sent to phone during session setup
3. Phone SHOULD render in centered sub-region with black bar margins
4. Shell.qml insets content area by `navbar.barThick` (UiMetrics.navbarThick)
5. VideoOutput fills inset content area with PreserveAspectCrop

### Recent SDP Changes (potential regression sources)
- `b8285ee` — Added `AdditionalVideoConfig` with `ui_theme=UI_THEME_AUTOMATIC` (ruled out)
- `5495551` — Added `color_scheme_support=COLOR_SCHEME_MATERIAL_YOU_V3` on AVChannel (ruled out)
- Proto submodule updated to v1.0 (`69733b8`) — field numbering may have shifted

### Secondary Issue: Navbar Thickness Mismatch
- C++ hardcodes 56px for margin calc and evdev zones
- QML uses `Math.round(56 * scale)` which differs when scale != 1.0
- Config default `ui.tokens.navbarThick = 0` means _tok() returns 0, but `isNaN(0)` is false so fallback doesn't trigger — however navbar IS visible on Pi so something overrides this

</code_context>
