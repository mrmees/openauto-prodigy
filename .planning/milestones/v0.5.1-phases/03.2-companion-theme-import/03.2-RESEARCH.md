# Phase 03.2: Companion Theme Import - Research

**Researched:** 2026-03-09
**Domain:** Qt 6 ThemeService expansion, TCP protocol message handling, M3 color system
**Confidence:** HIGH

## Summary

This phase extends the existing companion TCP socket protocol to receive Material Design 3 color palettes and wallpaper images from the companion Android app, expands ThemeService from 16 AA wire tokens to the full 34 M3 color roles, and adds companion theme lifecycle management (create, overwrite, delete, auto-switch).

The existing codebase provides strong foundations: CompanionListenerService already handles TCP connections with HMAC-SHA256 auth and message routing; ThemeService already supports theme scanning, YAML persistence, wallpaper resolution, and day/night color maps; the `applyAATokens` pattern demonstrates the color-application workflow. The expansion is primarily additive -- new Q_PROPERTYs, new message handlers, new theme persistence method -- with the AA wire token path being disabled (not deleted).

**Primary recommendation:** Structure implementation as three plans: (1) ThemeService 34-role expansion + bundled theme updates, (2) CompanionListenerService theme reception + protocol messages, (3) UI integration (theme deletion, AA wire disable).

## Architecture Patterns

### Current ThemeService Architecture (16 tokens)

The existing architecture stores colors in two flat `QMap<QString, QColor>` maps (`dayColors_`, `nightColors_`) keyed by hyphenated strings. Each color is exposed via a named Q_PROPERTY accessor that calls `activeColor("hyphenated-key")`. This pattern scales directly to 34 roles -- just add more Q_PROPERTYs and accessors.

**Current 16 base tokens:** primary, on-surface, surface, surface-variant, surface-container, surface-container-low, inverse-surface, inverse-on-surface, outline, outline-variant, background, text-primary, text-secondary, red, on-red, yellow, on-yellow

**Note:** `surface-container` exists as a key but has NO Q_PROPERTY accessor -- it's stored in YAML but not exposed to QML. This is a bug/gap in the current implementation that gets fixed naturally by the 34-role expansion.

This avoids breaking any QML that uses the old names. The YAML stores both sets of keys.

### Companion Theme Storage Pattern

```
~/.openauto/themes/
  midnight-blue/          # slug from theme name
    theme.yaml            # full 34-role palette
    wallpaper.jpg          # companion-provided wallpaper

```

**Theme YAML format for companion themes:**
```yaml
id: midnight-blue
name: Midnight Blue

seed: "#1A237E"            # M3 seed color (informational)

  primary: "#6750A4"

  primary-container: "#EADDFF"
  # ... all 34 roles

  primary: "#D0BCFF"
```

The `source: companion` field enables the settings UI to show a delete button only on companion-imported themes. Bundled themes have no `source` field.

### Slug Generation

Recommendation: lowercase, replace non-alphanumeric with hyphens, collapse multiple hyphens, trim.


### Chunk Reassembly

The design doc specifies base64-encoded JPEG chunks (~64KB each). The existing `QByteArray` approach in the design doc is correct:

1. `theme` message arrives: clear buffer, reserve expected size, store theme JSON
2. Each `theme_data` chunk: decode base64, append to buffer, increment counter
3. When `receivedChunks >= expectedChunks`: apply theme, save wallpaper, send ack
4. On disconnect: clear pending state

**Buffer management recommendation:** Cap maximum wallpaper size at 5MB (generous for a 1024x600 JPEG). Reject if `size` field exceeds cap. This prevents memory issues from malformed messages.

### Display Info in hello_ack

The `display` field needs the actual screen dimensions. Two sources:
1. `--geometry WxH` CLI flag (testing/dev) -- already parsed in main.cpp as `geomW`, `geomH`
2. `QGuiApplication::primaryScreen()->geometry()` (production)

The CompanionListenerService doesn't currently have access to either. Need to pass display dimensions to it, either via setter or by reading from DisplayInfo (already a global context property).

**Recommendation:** Add `setDisplaySize(int w, int h)` to CompanionListenerService. Wire it in main.cpp from the same source as DisplayInfo.

### AA Wire Token Disabling

The connection chain to disable:
1. `AndroidAutoOrchestrator.cpp:389` -- `connect(&videoHandler_, &VideoChannelHandler::uiConfigTokensReceived, ...)` calls `themeService_->applyAATokens()`
2. `ServiceDiscoveryBuilder.cpp:231` -- sets `color_scheme_support` field in SDP

**Recommendation:** Comment out or `#if 0` the connect in AndroidAutoOrchestrator. Leave the SDP field as-is (it doesn't hurt to advertise support even if we ignore the tokens). The `applyAATokens()` method stays in ThemeService but is simply never called from the AA protocol path. It can still be called by CompanionListenerService if the companion wants to apply a subset using the old 16-token format.

## Common Pitfalls

### Pitfall 1: Q_PROPERTY Declaration Explosion
**What goes wrong:** 34 base tokens + 5 derived = 39 Q_PROPERTYs in the header. Adding 18 more Q_PROPERTYs makes the header unwieldy.
**How to avoid:** Keep the pattern consistent -- each property is one-line Q_PROPERTY macro + one-line accessor method. Group by M3 category (primary group, secondary group, tertiary group, error group, surface group, etc.) with comments.

### Pitfall 2: YAML Key vs Q_PROPERTY Name Mismatch
**What goes wrong:** Accidentally using camelCase in YAML or hyphenated in Q_PROPERTY names.
**How to avoid:** Strict convention: YAML keys are hyphenated (`primary-container`), Q_PROPERTY names are camelCase (`primaryContainer`). The accessor calls `activeColor("primary-container")`.

### Pitfall 3: Bundled Theme M3 Derivation
**What goes wrong:** The 5 bundled themes only have 16 colors. Computing 34 M3 roles from them requires color science (tonal palettes from HCT color space).
**How to avoid:** Don't try to algorithmically derive M3 palettes. Manually pick reasonable values for each bundled theme's 18 new roles. The AMOLED theme is easy (many surface roles are pure black). For Default/Ocean/Ember, pick container variants that are lighter/darker than the base.

### Pitfall 4: Theme Scan Race with Companion Import
**What goes wrong:** Companion theme is saved to disk but `scanThemeDirectories()` hasn't been re-called, so the new theme doesn't appear in the picker.
**How to avoid:** After saving a companion theme, either: (a) call `scanThemeDirectories()` to refresh the full list, or (b) manually insert the new theme into `availableThemes_`/`availableThemeNames_`/`themeDirectories_` without a full rescan. Option (a) is simpler and the scan is fast (just reads YAML headers).

### Pitfall 5: Wallpaper Path Stale After Theme Delete
**What goes wrong:** Active wallpaper points to a file:// path inside a deleted theme directory.
**How to avoid:** `deleteTheme()` must check if the deleted theme is active, and if so, switch to "default" before deleting the directory.

### Pitfall 6: Partial M3 Palette from Companion
**What goes wrong:** Companion sends fewer than 34 roles (older app version, or intentionally sparse).
**How to avoid:** For any missing M3 role, fall back to a sensible default from the existing 16-token mapping or transparent. The `activeColor()` method already returns transparent for missing keys, and night mode falls back to day. This is probably fine -- QML elements bound to a missing property will get transparent (invisible), which is ugly but not a crash.

**Better approach:** Define a static fallback map that derives missing M3 roles from present ones (e.g., `on-primary` defaults to white if missing, `primary-container` defaults to primary with reduced opacity). Apply this during theme import, not at lookup time.

### Pitfall 7: theme_ack Must Include MAC
**What goes wrong:** Companion expects MAC-signed responses but `theme_ack` in the design doc is sent without MAC.
**How to avoid:** Check whether the companion app expects MAC on ack messages. Looking at the existing code, `hello_ack` is sent without MAC (pre-session). Status messages from the phone have MAC but the HU doesn't send status responses. The design doc shows `theme_ack` without MAC, which matches the pattern -- acks are unsigned.

### Theme Import Method (new, replaces connected-device specific logic)

```cpp
// Source: design doc + existing persistConnectedDeviceTheme pattern
bool ThemeService::importCompanionTheme(const QString& name, const QString& seed,
    const QMap<QString, QColor>& dayColors,
    const QMap<QString, QColor>& nightColors,
    const QByteArray& wallpaperJpeg)
{
    // Generate slug
    QString slug = name.toLower().trimmed();
    slug.replace(QRegularExpression("[^a-z0-9]+"), "-");
    slug.replace(QRegularExpression("^-|-$"), "");
    if (slug.isEmpty()) slug = "companion-theme";

    // Create/overwrite theme directory
    QString themeDir = QDir::homePath() + "/.openauto/themes/" + slug;
    QDir().mkpath(themeDir);

    // Write wallpaper
    if (!wallpaperJpeg.isEmpty()) {
        QFile wpFile(QDir(themeDir).filePath("wallpaper.jpg"));
        if (wpFile.open(QIODevice::WriteOnly | QIODevice::Truncate)) {
            wpFile.write(wallpaperJpeg);
        }
    }

    // Write theme YAML (all 34 roles)
    // ... yaml-cpp emitter pattern from persistConnectedDeviceTheme ...

    // Rescan to pick up the new/updated theme
    scanThemeDirectories(/* same search paths */);

    // Auto-switch
    setTheme(slug);
    // Clear wallpaper override so theme's own wallpaper is used
    setWallpaperOverride(QString());

    return true;
}
```

### Handling theme Message in CompanionListenerService

```cpp
// Source: design doc pattern
void CompanionListenerService::handleThemeMessage(const QJsonObject& msg)
{
    pendingThemeJson_ = msg["theme"].toObject();
    QJsonObject wp = msg["wallpaper"].toObject();
    expectedWallpaperChunks_ = wp["chunks"].toInt(0);
    int size = wp["size"].toInt(0);

    // Safety cap: 5MB max wallpaper
    if (size > 5 * 1024 * 1024) {
        qCWarning(lcCore) << "Companion: wallpaper too large:" << size;
        return;
    }

    receivedWallpaperChunks_ = 0;
    pendingWallpaperData_.clear();
    pendingWallpaperData_.reserve(size);
}
```

### Theme Deletion Method

```cpp
// New method on ThemeService
bool ThemeService::deleteTheme(const QString& themeId)
{
    // Prevent deleting bundled themes
    auto it = themeDirectories_.find(themeId);
    if (it == themeDirectories_.end()) return false;

    QString themeDir = it.value();
    // Check if it's a user theme (under ~/.openauto/themes/)
    QString userThemesDir = QDir::homePath() + "/.openauto/themes";
    if (!themeDir.startsWith(userThemesDir)) return false;

    // If this is the active theme, switch to default first
    if (themeId_ == themeId) {
        setTheme("default");
    }

    // Remove directory
    QDir(themeDir).removeRecursively();

    // Rescan
    scanThemeDirectories(/* search paths */);
    return true;
}
```

## Open Questions

1. **Search path persistence for rescan**
   - What we know: `scanThemeDirectories()` takes a `QStringList` of paths. After companion theme import, we need to rescan.
   - What's unclear: The search paths are constructed in `main.cpp` and not stored on ThemeService.
   - Recommendation: Store the search paths on ThemeService (set once during init) so `importCompanionTheme` and `deleteTheme` can trigger rescan internally.

2. **Existing QML usage of old property names**
   - What we know: All QML uses camelCase property names (textPrimary, onRed, etc.)
   - What's unclear: Whether any QML files use the color() method with string keys
   - Recommendation: Keep all old Q_PROPERTY names as-is. Add new M3 properties alongside. No QML breakage.

3. **scrim promotion from derived to stored**
   - What we know: `scrim()` currently returns hardcoded `QColor(0, 0, 0, 180)`. CONTEXT.md says promote to stored value.
   - What's unclear: M3 scrim is typically `#000000` (opaque black) -- the alpha comes from usage context, not the color token itself.
   - Recommendation: Store scrim as opaque `#000000` in YAML. The QML usage sites that need semi-transparent scrim should apply alpha themselves via `Qt.rgba(ThemeService.scrim.r, ThemeService.scrim.g, ThemeService.scrim.b, 0.7)`. This matches M3 spec.
