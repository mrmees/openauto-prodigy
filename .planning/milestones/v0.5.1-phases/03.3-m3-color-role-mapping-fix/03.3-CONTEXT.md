# Phase 03.3: M3 Color Role Mapping Fix - Context

**Gathered:** 2026-03-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Audit and correct the semantic mapping of M3 color roles to QML UI elements. After Phase 03.2 expanded ThemeService to 34 M3 roles, the QML layer still references legacy AA custom tokens (textPrimary, textSecondary, red, onRed, yellow) and uses only a narrow subset of the M3 palette. This phase remaps all QML color references to semantically correct M3 roles, removes deprecated custom tokens, adopts the surface container hierarchy for visual depth, and applies container/accent roles to interactive elements.

</domain>

<decisions>
## Implementation Decisions

### Text Color Identity
- Replace `textPrimary` (99 QML refs) with M3 `onSurface` — full removal, not alias
- Replace `textSecondary` (58 QML refs) with M3 `onSurfaceVariant` — full removal, not alias
- Remove `textPrimary` and `textSecondary` Q_PROPERTYs from ThemeService entirely
- Remove `text-primary` and `text-secondary` keys from all bundled theme YAMLs
- Remove from `knownAATokenKeys` set and connected-device persistence
- NormalText.qml default color changes from `textPrimary` to `onSurface`
- No new SubText control — secondary text color set per-use in QML
- Text on colored backgrounds uses M3 'on' counterparts (onPrimary, onError, etc.) for guaranteed contrast

### Error & Status Colors
- Replace `red` (15 QML refs) with M3 `error` — full removal
- Replace `onRed` (3 QML refs) with M3 `onError` — full removal
- Remove `red`/`onRed` keys from bundled theme YAMLs and knownAATokenKeys
- Use full error group: `error`/`onError` for destructive buttons, `errorContainer`/`onErrorContainer` for softer error contexts (banners, validation)
- `success`/`onSuccess` (17 QML refs) stays as custom derived color — no M3 equivalent, green is automotive-intuitive for "connected"/"confirm"
- `yellow` (3 QML refs) replaced with M3 `tertiary` — navbar accent icon color maps to complementary accent
- Remove `yellow`/`onYellow` keys from bundled theme YAMLs and knownAATokenKeys

### Surface Container Hierarchy
- Adopt M3 surface container levels mapped by visual elevation:
  - `surfaceContainerLowest` — deepest background (below cards)
  - `surfaceContainerLow` — cards, list items, button resting states
  - `surfaceContainer` — navbar, standard containers
  - `surfaceContainerHigh` — dialogs, elevated panels
  - `surfaceContainerHighest` — top-level overlays, popovers, gesture overlay
- Navbar changes from `surface` to `surfaceContainer` (M3 NavigationBar spec)
- Scrim uses M3 `scrim` role (already a Q_PROPERTY, 4 QML refs correct)

### Container & Accent Roles
- Button pressed states change from `primary` to `primaryContainer` with `onPrimaryContainer` text — softer, M3-standard contained look
- Remove `pressed` derived color from ThemeService — replaced by `primaryContainer`
- Toggle/slider active states change from `primary` fill to `secondaryContainer` — M3 uses secondary for selected/active toggle states
- Companion themes naturally provide distinct primary/secondary/tertiary container colors, making all these elements adapt

### Claude's Discretion
- Exact QML file-by-file mapping decisions (which specific elements get which container level)
- Whether `barShadow` derived color is still needed after surface hierarchy adoption
- How to handle edge cases where surfaceVariant is currently used (27 refs) — some may stay, some may remap to container levels
- Test coverage approach for verifying color role correctness

</decisions>

<specifics>
## Specific Ideas

- The goal is M3 compliance so companion themes "just work" — the phone's Material You palette should produce a cohesive car UI without manual tuning
- Automotive context means feedback needs to be obvious (pressed states visible at arm's length) but the primaryContainer approach is softer than raw primary — watch for visibility at the glanceability threshold
- Current usage of `surfaceContainerLow` for everything (26 refs) masks the elevation hierarchy — spreading across 5 levels will make companion themes significantly more expressive

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ThemeService` already exposes all 34 M3 roles + custom tokens as Q_PROPERTYs — this phase removes custom tokens, no new properties needed
- `NormalText.qml` control centralizes primary text color — single change propagates to many uses
- `colorsChanged` signal already triggers full QML rebinding — no additional wiring needed

### Established Patterns
- Q_PROPERTYs in camelCase (e.g., `onSurface`, `primaryContainer`) backed by hyphenated YAML keys (`on-surface`, `primary-container`)
- `activeColor(key)` lookup for current day/night value
- Derived colors computed in `recalcDerivedColors()` — `pressed` will be removed from there, `success`/`onSuccess`/`barShadow` stay

### Integration Points
- All 34+ QML files under `qml/` — bulk rename of color references
- `ThemeService.hpp/cpp` — remove 6 custom Q_PROPERTYs (textPrimary, textSecondary, red, onRed, yellow, onYellow), remove `pressed` derived
- `config/themes/*.yaml` — remove deprecated keys from all 5 bundled themes
- `ThemeService::allThemeKeys` set — remove deprecated entries
- `knownAATokenKeys` set — remove deprecated entries
- Connected-device theme persistence — remove deprecated keys from cache logic

</code_context>
