# Phase 03.3: M3 Color Role Mapping Fix - Research

**Researched:** 2026-03-09
**Domain:** Material Design 3 color system, QML theming, semantic color mapping
**Confidence:** HIGH

## Summary

This phase is a bulk QML refactoring task with targeted C++ changes. ThemeService already exposes all 34 M3 roles as Q_PROPERTYs -- the work is (1) removing 8 deprecated custom tokens from C++ and YAML, (2) remapping ~160 QML color references from legacy tokens to their M3 equivalents, and (3) spreading the flat `surfaceContainerLow` usage across the full 5-level surface container hierarchy.

The codebase is well-structured for this change. NormalText.qml centralizes primary text color (1 change propagates to all uses). The `colorsChanged` signal already triggers full QML rebinding -- no wiring changes needed. The main risk is subtle visual regressions (wrong "on" color paired with wrong container) that are only visible by running the app.

**Primary recommendation:** Execute as two waves -- Wave 1 removes deprecated tokens from C++ and does the bulk QML find-replace (textPrimary->onSurface, textSecondary->onSurfaceVariant, red->error, onRed->onError, yellow->tertiary). Wave 2 handles the nuanced surface container hierarchy remapping and button pressed-state changes.

### surfaceVariant Disposition (27 refs)

Most current `surfaceVariant` uses are for card/panel backgrounds and toggle unselected states. In the default theme, `surfaceVariant` equals `surface` (#16213e), making it visually redundant. Recommendation:

- **Toggle/segmented unselected state:** `surfaceContainerLow` (consistent button resting)
- **Card/panel backgrounds:** `surfaceContainerLow` (card elevation)
- **PhoneView dial pad:** `surfaceContainerLow` (consistent)
- **EQ bands/visualizers:** `surfaceContainerLow` (panel backgrounds)
- **NavbarPopup content area:** Keep `surfaceVariant` -- it's semantically correct for variant surfaces within a container

### barShadow Decision (Claude's Discretion)

**Recommendation: Keep `barShadow`.** It provides a 50% black border for navbar and power menu that creates consistent visual separation regardless of theme. The surface container hierarchy replaces background colors, not borders. Removing it would require finding another border strategy. Cost of keeping: 1 derived property, 2 QML refs. Not worth removing.

### Pressed State Pattern Change

Current pattern (appears ~20 times):
```qml
color: parent.pressed ? ThemeService.primary : ThemeService.surfaceContainerLow
```

New pattern:
```qml
color: parent.pressed ? ThemeService.primaryContainer : ThemeService.surfaceContainerLow
```

For text on pressed buttons, add explicit color binding:
```qml
// Text inside pressed button
color: parent.pressed ? ThemeService.onPrimaryContainer : ThemeService.onSurface
```

**Automotive visibility note:** `primaryContainer` is a lighter/softer variant of `primary`. In the default theme, day `primaryContainer` is `#f4a0b0` (light pink) vs `primary` `#e94560` (vivid red). This IS visibly different at arm's length. But companion themes could produce containers closer to the surface -- if glanceability becomes an issue post-implementation, the fallback is reverting to `primary` for pressed states only.

## Common Pitfalls

### Pitfall 1: Missing "on" Color Pairing
**What goes wrong:** Changing a background to `primaryContainer` but leaving text as `onSurface` -- text becomes unreadable on some themes.
**Why it happens:** M3 colors are designed in pairs (container + onContainer). Breaking the pairing breaks contrast guarantees.
**How to avoid:** Every time a background color changes, verify the text/icon color on top uses the matching "on" counterpart.
**Warning signs:** Text/icons that look fine in the default theme but disappear in companion themes.

### Pitfall 2: Hardcoded Colors Surviving the Refactor
**What goes wrong:** A `"transparent"`, `"white"`, `"#ffffff"`, or `Qt.darker()` call bypasses the theme system.
**Why it happens:** Easy to miss inline color values during bulk search-replace.
**How to avoid:** After refactoring, grep for remaining hardcoded color strings in QML.
**Warning signs:** Elements that don't change color when switching themes.

### Pitfall 3: Qt.darker/Qt.lighter Breaking with New Base Colors
**What goes wrong:** `Qt.darker(ThemeService.red, 1.15)` becomes `Qt.darker(ThemeService.error, 1.15)` -- but if `error` is already dark in a companion theme, `Qt.darker()` produces near-black.
**Why it happens:** `Qt.darker()` is relative -- on already-dark colors it over-darkens.
**How to avoid:** For pressed-state darkening of error buttons, use `errorContainer` instead of `Qt.darker(error)`. The M3 container variant IS the lighter/darker version.
**Warning signs:** Buttons that go nearly invisible when pressed in dark companion themes.

### Pitfall 4: Segmented Button Text Contrast
**What goes wrong:** Selected segment uses `secondaryContainer` background but text stays `textPrimary`/`onSurface` -- may not contrast.
**Why it happens:** `secondaryContainer` can be any tone; `onSurface` isn't guaranteed to contrast with it.
**How to avoid:** Selected text must use `onSecondaryContainer`, not `onSurface`.
**Warning signs:** Illegible selected option in segmented buttons.

### Pitfall 5: Build Failures from Removed Q_PROPERTYs
**What goes wrong:** QML references `ThemeService.textPrimary` but the Q_PROPERTY was removed from C++ -- runtime error (property returns undefined, text becomes invisible or black).
**Why it happens:** Q_PROPERTY removal in C++ doesn't cause compile errors in QML. It's a runtime binding failure.
**How to avoid:** The QML rename MUST happen in the same commit as the C++ removal. Run the app and verify every screen after.
**Warning signs:** `QML: Unable to assign [undefined] to QColor` warnings in console output.

### Pitfall 6: Connected-Device Theme YAML Stale Keys
**What goes wrong:** An existing `connected-device/theme.yaml` on the Pi still has `text-primary`, `red`, `yellow` keys. After removing them from `knownAATokenKeys` and `allThemeKeys`, they persist in the file but get filtered out during persistence, leading to silent color loss.
**Why it happens:** `persistConnectedDeviceTheme` filters through `allThemeKeys` -- removing keys from that set means existing values in the cached YAML won't round-trip.
**How to avoid:** This is actually the desired behavior -- old keys naturally age out. But verify that the connected-device theme still loads correctly with orphaned keys (it will -- `loadColorMap` doesn't filter, it loads all keys).

## Open Questions

1. **Companion theme container tone range**
   - What we know: Default theme has distinct container tones. Companion themes from Material You should too.
   - What's unclear: Whether some phone-generated palettes produce very close container tones that make the hierarchy invisible.
   - Recommendation: Not a blocker -- implement the hierarchy and tune if needed. Material You's tone algorithm is designed to produce distinguishable levels.

2. **Toggle/Switch Qt controls**
   - What we know: SettingsToggle uses Qt Quick Controls `Switch` which has its own styling.
   - What's unclear: Whether the Qt Switch respects ThemeService colors or has its own Material styling.
   - Recommendation: Check if the Switch needs explicit color overrides via `Material.accent`. If so, `main.qml` already sets `Material.accent: ThemeService.primary` -- may need to change to `secondaryContainer` for selected toggle states, but this could conflict with other Material-styled controls.
