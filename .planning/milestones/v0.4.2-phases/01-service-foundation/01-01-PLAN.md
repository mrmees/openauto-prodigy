  <action>
Add a new function `create_preflight_script()` to install.sh that generates `/usr/local/bin/openauto-preflight`. This script runs as ExecStartPre in the systemd service. It must:

1. **rfkill unblock** (self-healing):
   - `rfkill unblock wlan` and `rfkill unblock bluetooth`
   - Idempotent — safe to run when already unblocked
   - Print "PASS: WiFi radio unblocked" / "PASS: Bluetooth radio unblocked" on success
   - Exit non-zero if rfkill command fails (not just "already unblocked")

2. **WAYLAND_DISPLAY check:**
   - Check that `$XDG_RUNTIME_DIR/$WAYLAND_DISPLAY` socket exists (both env vars passed from service environment)
   - Print "PASS: Wayland compositor ready ($WAYLAND_DISPLAY)" or "FAIL: Wayland socket not found"
   - Exit non-zero on failure

3. **SDP socket check:**
   - Check `/var/run/sdp` exists and is group-writable by `bluetooth` group
   - Print "PASS: SDP socket ready" or "FAIL: /var/run/sdp missing or wrong permissions"
   - Exit non-zero on failure

Script structure: Each check prints a line. First failure exits non-zero. All pass = exit 0. Script should be useful standalone: `sudo openauto-preflight` prints human-readable results.

The script needs `#!/bin/bash` and should accept `--check-only` flag that skips the rfkill unblock (just reports state without self-healing). Use `set -euo pipefail`.

**Remove the one-time rfkill unblock from `configure_network()`** since it now runs on every start via pre-flight. Keep the BlueZ override ExecStartPost as-is (the pre-flight checks the result but doesn't replace the SDP socket permission logic).

Then **update `create_service()`** to generate a hardened unit file:

```ini
[Unit]
Description=OpenAuto Prodigy
After=graphical.target hostapd.service bluetooth.target pipewire.service
Wants=openauto-system.service
BindsTo=hostapd.service

[Service]
Type=notify
User=$USER
Environment=XDG_RUNTIME_DIR=/run/user/$USER_ID
Environment=WAYLAND_DISPLAY=wayland-0
Environment=QT_QPA_PLATFORM=wayland
Environment=DBUS_SESSION_BUS_ADDRESS=unix:path=/run/user/$USER_ID/bus
WorkingDirectory=$INSTALL_DIR
ExecStartPre=/usr/local/bin/openauto-preflight
ExecStart=$INSTALL_DIR/build/src/openauto-prodigy
ExecStopPost=-/bin/systemctl stop hostapd.service
Restart=on-failure
RestartSec=3
StartLimitBurst=5
StartLimitIntervalSec=60
WatchdogSec=30
NotifyAccess=main

[Install]
WantedBy=graphical.target
```

Key changes from current:
- `After=` adds hostapd, bluetooth, pipewire (ordering)
- `BindsTo=hostapd.service` — if hostapd dies, Prodigy is stopped. But during Prodigy crash restart, systemd restarts Prodigy without touching hostapd (BindsTo is one-directional: hostapd death kills Prodigy, but Prodigy death does NOT kill hostapd)
- `ExecStopPost=-/bin/systemctl stop hostapd.service` — on clean stop (exit 0 from `systemctl stop`), also stop the AP. The `-` prefix makes it non-fatal. On crash restart, ExecStopPost does NOT run (only runs on final stop, not between Restart= cycles)
- `Type=notify` + `NotifyAccess=main` — app sends sd_notify(READY=1)
- `WatchdogSec=30` — app must send WATCHDOG=1 within 30s
- `StartLimitBurst=5` + `StartLimitIntervalSec=60` — 5 crashes in 60s = failed state
- `ExecStartPre` runs the pre-flight script

Also ensure `create_preflight_script()` is called from the main install flow (where `create_service()` is called), before `create_service()`.

**Important:** The `ExecStopPost` to stop hostapd only runs after the service fully stops (not between restart cycles). This means: crash -> Restart=on-failure triggers -> hostapd stays up -> phone keeps WiFi. But `systemctl stop openauto-prodigy` -> ExecStopPost runs -> hostapd stops -> AP goes away. This is the behavior specified in CONTEXT.md.

Additionally, ensure `configure_network()` enables `systemd-networkd` and that the `10-openauto-ap.network` file includes `[Network] ConfigureWithoutCarrier=yes` so that systemd-networkd assigns the AP IP address even before hostapd brings the interface up. This ensures DHCP is ready before hostapd accepts clients (WIFI-04).
