## Phase Boundary

WiFi AP and system services start reliably on every boot with correct ordering and pre-conditions. Covers hostapd lifecycle management, systemd service ordering, crash recovery, and pre-flight checks. Bluetooth hardening (SDP registration resilience, --compat persistence) is Phase 2. App-level PipeWire readiness and clean shutdown is Phase 3.

<decisions>
## Implementation Decisions

### WiFi AP Lifecycle
- hostapd is tied to the Prodigy app lifecycle — starts when the app starts, stops when the app stops cleanly
- When Prodigy is not running, the Pi operates normally without the app interfering with WiFi or Bluetooth
- During crash restarts, hostapd stays up — phone keeps WiFi connection, only the app restarts
- Only a clean `systemctl stop` (or disable) takes down the AP
- rfkill unblock runs on every app start (pre-flight), not just at install time — survives unexpected rfkill state changes
- Strict ordering: systemd-networkd assigns AP IP (10.0.0.1) before hostapd begins accepting clients — no DHCP lease failures

### Startup Ordering
- Prodigy starts degraded and retries missing services — does not block waiting for all deps
- Faster to get something on screen; app handles missing BlueZ/PipeWire gracefully with retry logic
- Service type: `Type=notify` — app calls `sd_notify(READY=1)` after initialization complete; downstream services (web config) wait until actually ready
- `openauto-system` Python daemon is optional (`Wants=`, not `Requires=`) — core AA works without it
- Watchdog enabled: app sends periodic `sd_notify(WATCHDOG=1)` heartbeats; systemd kills and restarts on hang (>30s timeout catches deadlocks/freezes)

### Crash Recovery Policy
- Restart limit: 5 restarts in 60 seconds (`StartLimitBurst=5`, `StartLimitIntervalSec=60`)
- If limit exceeded, service enters failed state — something is fundamentally broken, stop trying
- Fixed 3-second restart delay (`RestartSec=3`)
- `Restart=on-failure` — clean shutdown (exit 0) stays stopped, crashes trigger restart
- No visual indicator on failure — journalctl/systemctl status is sufficient for debugging

### Pre-flight Checks
- ExecStartPre runs a standalone script (`/usr/local/bin/openauto-preflight`) — users can run it manually for debugging
- Checks: (1) rfkill wlan/bluetooth not soft-blocked, (2) WAYLAND_DISPLAY socket exists, (3) /var/run/sdp exists with correct permissions
- Self-healing: rfkill check actively unblocks (idempotent) rather than just detecting the problem
- Pre-flight failure blocks startup — service doesn't start, systemd retries after RestartSec
- Script is generated by install.sh alongside the service files

### Claude's Discretion
- Exact sd_notify integration points in the C++ app (which init steps constitute "ready")
- Watchdog heartbeat interval and timeout tuning
- Whether to use `sd_watchdog_enabled()` for dynamic interval or hardcode
- How to keep hostapd alive during crash restarts but stop it on clean shutdown (BindsTo= vs ExecStop= approach)
- Retry logic implementation for degraded startup (BlueZ/PipeWire reconnection patterns)
- Pre-flight script implementation details (exit codes, logging format)

</decisions>

<specifics>
## Specific Ideas

- The rfkill unblock in the pre-flight script covers both wlan AND bluetooth — single point of unblock for both radios
- The pre-flight script should be useful standalone: `sudo openauto-preflight` prints human-readable pass/fail for each check
- The BlueZ ExecStartPost SDP socket permission hack (5-retry sleep loop in install.sh) should be cleaned up as part of this
- When Prodigy stops cleanly, WiFi and BT should return to normal Pi behavior — no leftover hostapd or AP config active

</specifics>

### Established Patterns
- Service files generated as heredocs in install.sh functions — each service has its own create function
- systemd units go to `/etc/systemd/system/` — standard location, installer enables them
- Python system daemon already uses `Type=notify` — precedent for sd_notify integration
- App entry point in `src/main.cpp` — where sd_notify(READY=1) and watchdog heartbeat would be added

### Integration Points
- `install.sh` is the single source of truth for service file generation
- `src/main.cpp` — entry point for sd_notify integration
- BlueZ override at `/etc/systemd/system/bluetooth.service.d/override.conf` — SDP socket logic moves to pre-flight
- hostapd config at `/etc/hostapd/hostapd.conf` — already generated by installer
- hostapd needs lifecycle binding to Prodigy service (BindsTo= or ExecStartPost/ExecStopPost)
