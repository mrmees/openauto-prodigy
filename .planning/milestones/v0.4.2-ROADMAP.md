# Milestone v0.4.2: Service Hardening

**Status:** SHIPPED 2026-03-02
**Phases:** 1-3
**Total Plans:** 2 (formal) + direct commits for Phases 2-3

## Overview

Every lifecycle transition (install, first boot, reboot, app restart, crash recovery) must result in a fully working system with no manual intervention. Hardened the service stack from the bottom up: system services (WiFi AP, DHCP, systemd ordering), then Bluetooth/SDP initialization, then app lifecycle with clean startup/shutdown.

## Phases

### Phase 1: Service Foundation

**Goal**: WiFi AP and system services start reliably on every boot with correct ordering and pre-conditions
**Depends on**: Nothing (first phase)
**Plans**: 2 plans

Plans:
- [x] 01-01-PLAN.md — Pre-flight script and systemd service hardening (install.sh)
- [x] 01-02-PLAN.md — sd_notify integration and watchdog heartbeat (C++ app)

### Phase 2: Bluetooth Hardening

**Goal**: SDP registration and BlueZ compatibility work reliably across boots, daemon restarts, and package updates
**Depends on**: Phase 1

Delivered via direct commits (no formal GSD plans):
- Preflight script self-heals SDP socket permissions (group + mode check/fix)
- BlueZ --compat override via systemd drop-in (survives apt upgrade)
- App-side SDP retry with timer (2s intervals, max retries, clear logging)

### Phase 3: App Lifecycle

**Goal**: The app starts cleanly (waiting for PipeWire), shuts down cleanly (no orphaned resources), and the installer verifies everything works
**Depends on**: Phase 1, Phase 2

Delivered via existing implementation + Phase 1 work:
- AudioService graceful degradation (pw_context_connect failure → isAvailable()=false → createStream returns nullptr)
- Destructor destroys all PW streams under lock (capture + playback + device registry + core)
- main.cpp shutdown: pluginManager.shutdownAll() + bluetoothManager->shutdown() + sd_notify STOPPING
- Installer run_diagnostics: checks PipeWire, BlueZ, hostapd, labwc config, group memberships

## Milestone Summary

**Key Decisions:**
- Pre-flight script generated by installer heredoc (single source of truth)
- BindsTo=hostapd is one-directional — hostapd death stops Prodigy, not vice versa
- PartOf= instead of ExecStopPost for hostapd lifecycle
- libsystemd is optional (QUIET) — builds without it on non-systemd systems
- Watchdog interval dynamically read from sd_watchdog_enabled with 15s fallback
- All sd_notify code uses #ifdef HAS_SYSTEMD, matching existing HAS_BLUETOOTH pattern

**Issues Resolved:**
- WiFi AP dying when Prodigy app restarts
- SDP socket permission races after BlueZ restart
- wf-panel-pi BT popup drawing over Prodigy UI
- HAS_SYSTEMD flag applied to wrong CMake target

**Technical Debt:** None significant.
