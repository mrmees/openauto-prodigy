  <name>Task 2: Run four-point proof chain — negative baseline then positive HTTP+HTTPS</name>
  <files>docs/validation/phase18-hardware-validation-TIMESTAMP.log</files>
  <action>
    All work via SSH to matt@192.168.1.152. Capture all commands and output to a timestamped log file on the Pi at `/tmp/phase18-validation-$(date +%Y%m%d-%H%M%S).log`.

    **Step 0: Pre-flight checks**
    - Confirm openauto-system service is active: `systemctl is-active openauto-system.service`
    - Detect active outbound interface programmatically: `ip route get 8.8.8.8 | grep -oP 'dev \K\S+'`
    - Identify redsocks redirect port from config or process listing
    - Check what companion app logs via ADB: `adb logcat -d | grep -i 'socks\|proxy\|forward'` — verify observability exists BEFORE relying on it. If companion does not emit clear SOCKS forwarding logs, document the gap and note that proof point 3 (companion receipt) cannot be satisfied.

    **Step 1: Negative case (proxy DISABLED)**
    - Ensure proxy is disabled: call `set_proxy_route(active=false)` via IPC or the Python client
    - Verify `get_proxy_status` reports DISABLED
    - Run: `curl -s http://httpbin.org/ip` — should succeed via direct path
    - Record result. This establishes the baseline: traffic flows directly, not through companion.

    **Step 2: Positive case — enable proxy**
    - Call `set_proxy_route(active=true)` via IPC
    - Verify `get_proxy_status` reports ACTIVE
    - If not ACTIVE, record the actual status and error detail but continue collecting evidence

    **Step 3: Positive HTTP test (httpbin.org/ip) — four-point proof**
    Run these in parallel (background tcpdump captures, then curl):

    1. **Pi redirect proof (loopback):** `sudo tcpdump -i lo -c 20 -w /tmp/phase18-lo-http.pcap port {redirect_port} &` — start capture, then run curl, then stop capture. Check if packets were captured: `tcpdump -r /tmp/phase18-lo-http.pcap | head -20`
    2. **Pi bypass proof (outbound):** `sudo tcpdump -i {outbound_iface} -c 10 -w /tmp/phase18-out-http.pcap host httpbin.org &` — start capture, run curl, stop capture. Verify NO packets to httpbin.org on outbound interface (traffic should go to companion instead).
    3. **Companion receipt:** `adb logcat -d | grep -i 'socks\|proxy\|forward\|httpbin'` — check for companion-side evidence of the SOCKS connection/forwarding. If companion logging was found insufficient in Step 0, skip and note as unverifiable.
    4. **Request success:** `curl -s --max-time 15 http://httpbin.org/ip` — should return JSON with an IP (may be the companion/phone's IP rather than Pi's direct IP).

    Record pass/fail for each proof point with captured output.

    **Step 4: Positive HTTPS test (google.com) — four-point proof**
    Same four-point pattern as Step 3 but with `https://www.google.com/`:

    1. **Pi redirect proof:** tcpdump on lo for redirect port
    2. **Pi bypass proof:** tcpdump on outbound for google.com (port 443)
    3. **Companion receipt:** logcat grep for google/SOCKS evidence
    4. **Request success:** `curl -s --max-time 15 -o /dev/null -w '%{http_code}' https://www.google.com/` — should return 200

    Record pass/fail for each proof point.

    **Step 5: Disable and cleanup**
    - Call `set_proxy_route(active=false)`
    - Verify `get_proxy_status` reports DISABLED
    - Clean up pcap files

    **Step 6: Verdict**
    - Tally all proof points (negative baseline + 2x4 positive proof points = 9 checks)
    - Overall PASS only if all required proof points succeed
    - If companion logging was insufficient, overall verdict is BLOCKED with documented gap
    - Write verdict to the log file

    **Step 7: Copy validation log to repo**
    - `scp matt@192.168.1.152:/tmp/phase18-validation-*.log docs/validation/`
    - Create `docs/validation/` directory if it does not exist
    - Commit the validation log to git

    **Failure handling:** Do NOT stop on individual proof-point failure. Continue collecting all evidence. The full picture is needed for diagnosis.
