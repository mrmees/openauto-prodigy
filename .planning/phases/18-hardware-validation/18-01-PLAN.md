  <name>Task 2: Run four-point proof chain — negative baseline then positive HTTP+HTTPS</name>
  <files>docs/validation/phase18-hardware-validation-TIMESTAMP.log</files>
  <action>
    Pi-side work via SSH to matt@192.168.1.152. Companion-side evidence via ADB from the claude-dev VM (ADB is on the VM, not the Pi). Set `TIMESTAMP=$(date +%Y%m%d-%H%M%S)` at the start and use it consistently. Capture all commands and output to `/tmp/phase18-validation-${TIMESTAMP}.log` on the Pi.

    **Step 0: Pre-flight checks**
    - Confirm openauto-system service is active: `systemctl is-active openauto-system.service`
    - Detect active outbound interface programmatically: `ip route get 8.8.8.8 | sed -n 's/.*dev \([^ ]*\).*/\1/p'` (no `grep -oP` — unavailable on minimal Trixie)
    - Identify redsocks redirect port from config or process listing
    - Check what companion app logs via ADB **from the claude-dev VM** (not over SSH — ADB is on the VM, not the Pi): `adb logcat -c` (clear buffer first), then trigger a test SOCKS request from the Pi over SSH, then `adb logcat -d | grep -i 'socks\|proxy\|forward'` on the VM. This establishes whether the companion emits usable SOCKS forwarding logs. If it does not, document the gap and note that proof point 3 (companion receipt) cannot be satisfied.
    - **Execution model note:** Steps that check companion logs via ADB run from the claude-dev VM. All other steps (tcpdump, curl, IPC calls) run over SSH to the Pi. The executor must interleave VM-local and SSH commands.

    **Step 1: Negative case (proxy DISABLED)**
    - Ensure proxy is disabled: call `set_proxy_route(active=false)` via IPC or the Python client
    - Verify `get_proxy_status` reports DISABLED
    - Clear companion logcat buffer from the VM: `adb logcat -c`
    - Run on Pi over SSH: `curl -s http://httpbin.org/ip` — should succeed via direct path
    - Check companion logs from the VM: `adb logcat -d | grep -i 'socks\|proxy\|forward\|httpbin'` — should show NO SOCKS forwarding events for this request, proving traffic went direct
    - Record both results (curl success + absence of companion SOCKS evidence). This establishes the baseline: with proxy off, traffic flows direct and companion does not see it.

    **Step 2: Positive case — enable proxy**
    - Call `set_proxy_route(active=true)` via IPC
    - Verify `get_proxy_status` reports ACTIVE
    - If not ACTIVE, record the actual status and error detail but continue collecting evidence

    **Step 3: Positive HTTP test (httpbin.org/ip) — four-point proof**
    Resolve target IP before capture to prevent DNS mismatch between tcpdump filter and curl:
    - On Pi: `HTTP_TARGET_IP=$(dig +short httpbin.org | head -1)` — pin the resolved IP for both tcpdump and curl verification

    Run captures then curl:

    1. **Pi redirect proof (loopback):** `sudo tcpdump -i lo -c 20 -w /tmp/phase18-lo-http.pcap port {redirect_port} &` — start capture, then run curl, then stop capture. Check if packets were captured: `tcpdump -r /tmp/phase18-lo-http.pcap | head -20`
    2. **Pi bypass proof (outbound):** `sudo tcpdump -i {outbound_iface} -c 10 -w /tmp/phase18-out-http.pcap host $HTTP_TARGET_IP &` — use resolved IP, not hostname. Start capture, run curl, stop capture. Verify NO packets to the target IP on outbound interface (traffic should go to redsocks/companion instead).
    3. **Companion receipt:** Clear logcat from VM (`adb logcat -c`), run curl on Pi, then check from VM: `adb logcat -d | grep -i 'socks\|proxy\|forward\|httpbin'` — scoped to events after the clear. If companion logging was found insufficient in Step 0, skip and note as unverifiable.
    4. **Request success:** `curl -s --max-time 15 http://httpbin.org/ip` — should return JSON with an IP (may be the companion/phone's IP rather than Pi's direct IP).

    Record pass/fail for each proof point with captured output.

    **Step 4: Positive HTTPS test (google.com) — four-point proof**
    Same four-point pattern as Step 3 but with `https://www.google.com/`:
    - Resolve first: `HTTPS_TARGET_IP=$(dig +short www.google.com | head -1)`

    1. **Pi redirect proof:** tcpdump on lo for redirect port
    2. **Pi bypass proof:** tcpdump on outbound for `$HTTPS_TARGET_IP` (use resolved IP, not hostname)
    3. **Companion receipt:** Clear logcat (`adb logcat -c` on VM), run curl on Pi, then check logcat from VM for google/SOCKS evidence
    4. **Request success:** `curl -s --max-time 15 -o /dev/null -w '%{http_code}' https://www.google.com/` — should return 200

    Record pass/fail for each proof point.

    **Step 5: Disable and cleanup**
    - Call `set_proxy_route(active=false)`
    - Verify `get_proxy_status` reports DISABLED
    - Clean up pcap files

    **Step 6: Verdict**
    - Tally all proof points: negative baseline (2 checks: curl success + no companion receipt) + 2x4 positive proof points = 10 checks
    - Overall PASS only if all required proof points succeed
    - If companion logging was insufficient, overall verdict is BLOCKED with documented gap
    - Write verdict to the log file with the specific `VALIDATION_LOG` filename used throughout

    **Step 7: Copy validation log to repo**
    - Use the specific timestamped filename (not a wildcard): `scp matt@192.168.1.152:/tmp/phase18-validation-${TIMESTAMP}.log docs/validation/phase18-hardware-validation-${TIMESTAMP}.log`
    - Create `docs/validation/` directory if it does not exist
    - Commit the specific file to git (not a glob)

    **Failure handling:** Do NOT stop on individual proof-point failure. Continue collecting all evidence. The full picture is needed for diagnosis.
