# Phase 13.1: Quick Bugfix for Companion App - Context

**Gathered:** 2026-03-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix stale CompanionListenerService client blocking all future companion reconnects. A single shared cleanup path, always-replace on new connections, socket error handling, and inactivity timeout. No UI changes — log-only diagnostics.

</domain>

<decisions>
## Implementation Decisions

### Stale detection strategy
- **Always-replace:** When a new connection arrives and `client_` is non-null, abort the old socket and accept the new one
- Log old peer address/port, new peer address/port, and reason (`replaced_by_new_connection`) before replacing
- Reuse the same full cleanup path as disconnect — clear `sessionKey_`, `lastSeq_`, GPS/battery/internet state, proxy route, pending theme transfer state
- Only one companion client is expected; newest connection always wins

### Socket error handling
- Connect `QTcpSocket::errorOccurred` signal to the same cleanup path
- Log the socket error and peer before cleanup
- Cleanup must be idempotent — `errorOccurred` followed by `disconnected` must not double-free or double-clear

### Inactivity timeout
- 30-second timeout from last **valid authenticated status message** (not just any socket activity)
- Treat timeout as session expiry — full cleanup (client_, session key, proxy route, published companion state)
- Feed timeout into the same shared teardown path as disconnected and errorOccurred
- Companion app sends status every ~5s, so 30s = 6 missed intervals

### Shared cleanup path
- Extract one idempotent `clearClientSession()` method from current `onClientDisconnected()` logic
- All teardown triggers route through it: disconnect, errorOccurred, inactivity timeout, always-replace
- Current `onClientDisconnected()` body moves into `clearClientSession()`

### UI / Logging
- No UI changes — existing `connectedChanged` Q_PROPERTY signal handles connected/disconnected transitions naturally
- Log-only diagnostics on the replacement path: old peer, new peer, reason
- Log on timeout expiry and socket errors

### Claude's Discretion
- QTimer vs other mechanism for inactivity tracking
- Exact log format and verbosity levels (qCInfo vs qCWarning)
- Whether to also guard against nullptr in cleanup edge cases

</decisions>

<specifics>
## Specific Ideas

- The cleanup function must clear ALL session state: sessionKey_, lastSeq_, GPS, battery, internet, proxy route, pending theme transfer, and emit all changed signals
- Timer should reset only on valid MAC-verified status messages, not on hello or theme messages
- The bug report notes there's no log line on the current reject path — fix that regardless

</specifics>
