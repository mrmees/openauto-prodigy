# Phase 13.2: Fix saving of themes received by companion app - Context

**Gathered:** 2026-03-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix two separate issues: (1) **Bug:** `ThemeService::setTheme()` doesn't persist to config.yaml, so companion-imported themes revert on restart; (2) **UX hardening:** the wallpaper override picker is too easy to accidentally change, masking a theme's own wallpaper. These are independent problems — the first is a persistence bug, the second is adjacent UX improvement that was mistaken for the root cause.

</domain>

<decisions>
## Implementation Decisions

### setTheme() config persistence
- `ThemeService::setTheme()` must persist the theme ID to config
- Give ThemeService an `IConfigService*` pointer (set in main.cpp like other services)
- `setTheme()` calls `config->setValue("display.theme", slug)` + `config->save()` after loading the theme
- This fixes ALL callers (companion import, settings UI, any future caller) — not just companion import

### Wallpaper override toggle gate
- Add a "Custom Wallpaper" toggle to ThemeSettings.qml
- When OFF (default): wallpaper comes from theme, picker is hidden
- When ON: wallpaper override picker becomes visible, user can select a custom wallpaper
- Toggle OFF clears any existing wallpaper override (resets to theme default immediately)
- Toggle state: derived from `display.wallpaper_override` being non-empty (empty string = OFF/theme default). Note: the existing "None" override (`"none"`) is an intentional override that means "no wallpaper at all" — this should show as toggle ON with "None" selected in the picker, not as toggle OFF

### Delete theme button placement
- Move the "Delete Theme" button/row to the bottom of the theme settings list (currently row 1, right under theme picker)
- Separation from the theme picker prevents accidental taps

### Claude's Discretion
- Exact toggle component (SettingsToggle or custom)
- Animation/transition when showing/hiding the wallpaper picker
- Whether to remove "Theme Default" from the wallpaper picker options when the toggle implies it
- Whether to strip "None" from the picker (since toggle OFF already means theme default)

</decisions>

<specifics>
## Specific Ideas

- The toggle state should be computed from `wallpaper_override` — empty string = OFF (theme default), any non-empty value including "none" = ON (active override). "None" is a deliberate "no wallpaper" choice, not the same as "use theme default."
- The original bug report was "themes received by companion app don't persist" — the actual root cause is `setTheme()` not persisting to config. The wallpaper picker UX is a separate improvement that prevents users from accidentally masking a theme's wallpaper.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `SettingsToggle` component: already used for "Always Use Dark Mode" in ThemeSettings.qml
- `FullScreenPicker` component: used for both Theme and Wallpaper pickers
- `ConfigService.onConfigChanged(path, value)`: signal for reacting to config changes in QML
- `YamlConfig::setTheme()` / `YamlConfig::theme()`: config read/write methods already exist

### Established Patterns
- Settings controls use `configPath` property for auto-persistence via `ConfigService.setValue()`
- `ThemeService` is set on HostContext and available to all plugins/QML
- Service pointers injected via setters in main.cpp; other services already take `IConfigService*` (e.g. BluetoothManager, AndroidAutoOrchestrator)

### Integration Points
- `ThemeService::setTheme()` (line 181): needs `IConfigService*` to call `setValue("display.theme", slug)` + `save()`
- `ThemeSettings.qml` (line 109-121): wallpaper picker row needs toggle gate
- `ThemeSettings.qml` (line 34-107): delete theme row needs to move to bottom
- `main.cpp` (line 200-204): theme loading on startup, needs to also wire ConfigService to ThemeService
- `importCompanionTheme()` calls `setTheme(slug)` — will auto-persist once setTheme() is fixed

</code_context>
