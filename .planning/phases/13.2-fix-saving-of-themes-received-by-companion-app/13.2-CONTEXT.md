# Phase 13.2: Fix saving of themes received by companion app - Context

**Gathered:** 2026-03-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix theme persistence from companion app and harden the theme settings UX to prevent accidental wallpaper override. Two bugs: (1) `ThemeService::setTheme()` doesn't persist to config.yaml, so companion-imported themes revert on restart; (2) the wallpaper override picker is too easy to accidentally change, masking the theme's own wallpaper.

</domain>

<decisions>
## Implementation Decisions

### setTheme() config persistence
- `ThemeService::setTheme()` must persist the theme ID to config via `ConfigService`
- Give ThemeService a `ConfigService*` pointer (set in main.cpp like other services)
- `setTheme()` calls `config->setTheme(slug)` + `config->save()` after loading the theme
- This fixes ALL callers (companion import, settings UI, any future caller) — not just companion import

### Wallpaper override toggle gate
- Add a "Custom Wallpaper" toggle to ThemeSettings.qml
- When OFF (default): wallpaper comes from theme, picker is hidden
- When ON: wallpaper override picker becomes visible, user can select a custom wallpaper
- Toggle OFF clears any existing wallpaper override (resets to theme default immediately)
- Config path: use existing `display.wallpaper_override` — empty string = theme default, so toggle OFF writes empty string

### Delete theme button placement
- Move the "Delete Theme" button/row to the bottom of the theme settings list (currently row 1, right under theme picker)
- Separation from the theme picker prevents accidental taps

### Claude's Discretion
- Exact toggle component (SettingsToggle or custom)
- Animation/transition when showing/hiding the wallpaper picker
- Whether the toggle state is derived from `wallpaper_override` being non-empty (computed) vs a separate config key

</decisions>

<specifics>
## Specific Ideas

- The toggle state should probably be computed from `wallpaper_override` — if it's non-empty and not "none", the toggle is ON. No need for a separate config key.
- The original bug report was "themes received by companion app don't persist" — root cause is `setTheme()` not writing to config, plus the wallpaper picker making it easy to accidentally override the companion theme's wallpaper.

</specifics>

## Existing Code Insights

### Reusable Assets
- `SettingsToggle` component: already used for "Always Use Dark Mode" in ThemeSettings.qml
- `FullScreenPicker` component: used for both Theme and Wallpaper pickers
- `ConfigService.onConfigChanged(path, value)`: signal for reacting to config changes in QML
- `YamlConfig::setTheme()` / `YamlConfig::theme()`: config read/write methods already exist

### Established Patterns
- Settings controls use `configPath` property for auto-persistence
- `ThemeService` is set on HostContext and available to all plugins/QML
- Service pointers injected via setters in main.cpp (setConfigService, setThemeService, etc.)

### Integration Points
- `ThemeService::setTheme()` (line 181): needs ConfigService pointer to persist
- `ThemeSettings.qml` (line 109-121): wallpaper picker row needs toggle gate
- `ThemeSettings.qml` (line 34-107): delete theme row needs to move to bottom
- `main.cpp` (line 200-204): theme loading on startup, needs to also wire ConfigService to ThemeService
- `importCompanionTheme()` calls `setTheme(slug)` — will auto-persist once setTheme() is fixed
