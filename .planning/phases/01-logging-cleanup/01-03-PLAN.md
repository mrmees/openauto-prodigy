  <action>
Audit each file and downgrade repetitive per-operation messages from qCInfo to qCDebug. The triage rule:

**Keep as qCInfo:** One-time events, state transitions, connection established/lost, session start/end, errors recovered from, configuration applied. Things a user running in quiet mode SHOULD see.

**Downgrade to qCDebug:** Per-packet, per-frame, per-attempt, per-property-change, per-poll, hex dumps, buffer stats, retry attempts, slot-level touch events. Things only useful when actively debugging.

Per-file guidance from the UAT diagnosis:

- **BluetoothManager.cpp** (~38 log calls): Per-attempt auto-connect retries, per-property-change D-Bus updates, per-device scan results -> qCDebug. Keep: device connected/disconnected, pairing success/fail, agent registered.

- **AndroidAutoOrchestrator.cpp** (~22 log calls): Per-watchdog-poll TCP state checks, per-step lifecycle detail -> qCDebug. Keep: orchestrator started/stopped, connection established/lost, session begin/end.

- **EvdevTouchReader.cpp** (~15 log calls): Per-slot DOWN/UP/MOVE events, resolution updates, ABS values -> qCDebug. Keep: device opened/closed, grab/ungrab state changes, 3-finger gesture detected.

- **BluetoothDiscoveryService.cpp** (~21 log calls): Hex dumps of SDP/RFCOMM data, per-message proto parsing, per-byte wire format -> qCDebug. Keep: discovery started/completed, device found, connection success/fail.

- **VideoDecoder.cpp** (~13 log calls): Per-packet codec sniffing, per-frame decode stats, buffer pool stats -> qCDebug. Keep: decoder initialized, codec selected, hw/sw fallback decision, decoder error.

- **AudioService.cpp** (~9 log calls): Per-stream buffer growth, ring buffer stats -> qCDebug. Keep: stream created/destroyed, audio focus changes, PipeWire connection.

After changes, verify the ratio has flipped: qCDebug should be the MAJORITY of calls, qCInfo the minority (one-time events only).
