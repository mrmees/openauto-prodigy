**Two-stage snap threshold recovers gutter waste (7x3 to 7x4 on 1024x600) with tighter card padding (spacing/4 vs spacing/2)**

## Accomplishments
- Replaced fixed grid frame computation with snap-aware two-stage version that adds rows/cols when waste exceeds 60% of base cell size

- Card padding halved from spacing/2 (~4px) to spacing/4 (~2px, minimum 2px floor) for tighter visual grid
- Codex review completed; P2 finding (post-snap column fit) acknowledged as working-as-designed (single-pass threshold is intentional)

## Files Created/Modified

## Decisions Made
- Single-pass snap threshold: snap is evaluated once against baseCellSide, not iteratively against the shrunk effective cellSide. This prevents cascading shrinkage on displays where post-snap cell reduction opens space for more cols/rows. The 7x4 target on 1024x600 matches research analysis.
- kSnapThreshold = 0.6 as a readonly QML property (not YAML-configurable). The value was analytically chosen; user tuning adds complexity for no benefit.
- Card padding minimum of 2px to prevent glass card radius clipping artifacts on the Pi's 1024x600 display.

None - plan executed exactly as written.

## Issues Encountered

- Codex review raised P2 finding that post-snap cellSide (120.25px) allows an 8th column to fit (8*120.25=962 < 984px pageView width). Acknowledged as working-as-designed: the snap threshold operates on baseCellSide (131.9px), where X-axis waste was 46% (below 60% threshold). Iterative re-evaluation was not the intended algorithm.
