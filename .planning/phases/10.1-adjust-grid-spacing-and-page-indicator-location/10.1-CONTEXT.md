# Phase 10.1: Adjust Grid Spacing and Page Indicator Location - Context

**Gathered:** 2026-03-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Move the page indicator dots from the HomeMenu ColumnLayout into the navbar (flanking the clock), and adjust the grid cell sizing in QML to reduce gutter waste via an auto-snap threshold. Remove the PageIndicator block to reclaim vertical space.

</domain>

<decisions>
## Implementation Decisions

### Page indicator relocation
- **Move page dots from HomeMenu into the navbar**, flanking the clock control
- **The clock IS the active page indicator** ŌĆö it replaces the "active" dot. Inactive pages show as small dots on either side of the clock
- For 3 pages on page 0: `[Clock ŌŚŗ ŌŚŗ]`. On page 1: `[ŌŚŗ Clock ŌŚŗ]`. On page 2: `[ŌŚŗ ŌŚŗ Clock]`
- **Dot styling:** Same small dots as current (primary color for active position is the clock itself, dim onSurfaceVariant for inactive dots)
- **Visual only** ŌĆö dots are NOT tappable. This is an intentional behavior change from the current tappable `PageIndicator` (HomeMenu.qml:538/553). Dots are too small in the navbar to be reliable tap targets at car speeds. Users swipe in the grid to change pages.
- **Reserved page counts** ŌĆö the reserved page (Settings/AA) gets a dot like any other page. Users should know it exists and where they are relative to it.
- **Hide when single page** ŌĆö if only 1 page exists (just the reserved page), no dots appear. The clock stands alone. Dots appear when there are 2+ total pages (including the reserved page).
- **Remove `PageIndicator` from HomeMenu.qml** ŌĆö the entire `PageIndicator` block (line ~533) and its ColumnLayout slot are deleted. This reclaims vertical space for the grid.
- **Vertical navbar layout** ŌĆö when navbar is on left/right edge (Navbar.qml:9, vertical mode at NavbarControl.qml:95), dots stack vertically above/below the clock instead of flanking horizontally. The clock-as-active-indicator concept works the same way, just rotated.

### Grid cell auto-snap threshold
- **Current problem:** `cellSide = diagPx / 9.0` produces cells that leave large gutters when the available space is "almost" enough for another row/col. On 1024x600, this wastes 88px vertically (2/3 of a cell).
- **Auto-snap logic MUST live in QML (HomeMenu.qml)**, not DisplayInfo.cpp. DisplayInfo only knows window diagonal and bias ŌĆö it does NOT know the post-layout usable rect, margins, or navbar edge. This was explicitly called out in Phase 09 research (09-RESEARCH.md:248) and the grid frame ownership split (09-CONTEXT.md:51). DisplayInfo.cellSide remains the ideal/baseline cell size; QML applies snap on top of it.
- **Auto-snap rule:** After computing initial `gridCols` and `gridRows` from `floor(available / cellSide)`:
  - Calculate leftover for each axis: `xWaste = availW - gridCols * cellSide`, `yWaste = availH - gridRows * cellSide`
  - If `xWaste > 0.6 * cellSide`, increment `gridCols` by 1
  - If `yWaste > 0.6 * cellSide`, increment `gridRows` by 1
  - Then recompute effective cell size to fit the new grid dims: `effectiveCellSide = min(availW / gridCols, availH / gridRows)` (cells stay square)
- **60% threshold** ŌĆö conservative enough to avoid micro-cells, aggressive enough to catch the 82% waste case on 1024x600
- **Floor minimums still apply:** 3 cols / 2 rows minimum regardless of snap

### Cell gap and gutter handling
- **Keep geometry unchanged ŌĆö tighten internal padding, not inter-cell pitch.** Current layout math assumes `pitch == cellSide` throughout HomeMenu.qml (lines 23, 207, 232). Introducing a real inter-cell gap would require rewriting pitch math in delegate positioning, drag/resize coordinate mapping, and highlight placement. Instead: reduce the existing internal card padding (HomeMenu.qml:280, `anchors.margins: UiMetrics.spacing / 2`) to a smaller value so widgets appear closer together visually without changing the grid coordinate system.
- **Remaining gutter** after snap absorbed into symmetric edge margins (`offsetX`/`offsetY`) ŌĆö same pattern as today, just less waste from the snap threshold
- **No new pitch math, no inter-cell gap constant** ŌĆö the snap threshold alone recovers most of the wasted space

### Navbar height ŌĆö NO additional deduction
- **Do NOT subtract navbar thickness from grid available height.** Shell.qml (lines 14, 17) already lays out HomeMenu in a content rect with navbar space removed. The grid's `pageView.width`/`pageView.height` already reflects the post-navbar available space. Subtracting `navbarThick` again would double-shrink the grid.
- This preserves the ownership split from Phase 09 (09-CONTEXT.md:51): Shell owns layout boundaries, HomeMenu owns grid frame within its given rect.

### Claude's Discretion
- Internal card padding reduction amount (from `UiMetrics.spacing / 2` to something tighter)
- How the navbar communicates page state to display dots (signal from WidgetGridModel or direct QML binding via context property)
- Navbar dot layout implementation (inline in NavbarControl, separate component, or extension of the clock control)
- Whether the snap threshold value (0.6) should be a constant in QML or configurable

</decisions>

## Specific Ideas

- "Page indicators are taking up too much usable space ŌĆö would like to see them on either side of the clock in the navbar instead"
- "The clock should be the indicator of the page you're on. If on the home page and 3 pages exist, two dots to the right. If on the settings page, two dots to the left."
