# Phase 10.1: Adjust Grid Spacing and Page Indicator Location - Context

**Gathered:** 2026-03-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Move the page indicator dots from the HomeMenu ColumnLayout into the navbar (flanking the clock), and adjust the grid cell sizing formula to reduce gutter waste via an auto-snap threshold. Also account for navbar thickness in available grid height computation.

</domain>

<decisions>
## Implementation Decisions

### Page indicator relocation
- **Move page dots from HomeMenu into the navbar**, flanking the clock control
- **The clock IS the active page indicator** — it replaces the "active" dot. Inactive pages show as small dots on either side of the clock
- For 3 pages on page 0: `[Clock ○ ○]`. On page 1: `[○ Clock ○]`. On page 2: `[○ ○ Clock]`
- **Dot styling:** Same small dots as current (primary color for active position is the clock itself, dim onSurfaceVariant for inactive dots)
- **Visual only** — dots are NOT tappable. Users swipe in the grid to change pages.
- **Hide when single page** — if only 1 page exists, no dots appear. The clock stands alone. Dots only appear when there are 2+ pages.
- **Remove `PageIndicator` from HomeMenu.qml** — the entire `PageIndicator` block (line ~533) and its ColumnLayout slot are deleted. This reclaims vertical space for the grid.

### Grid cell auto-snap threshold
- **Current problem:** `cellSide = diagPx / 9.0` produces cells that leave large gutters when the available space is "almost" enough for another row/col. On 1024x600, this wastes 88px vertically (2/3 of a cell).
- **Auto-snap rule:** After computing initial `gridCols` and `gridRows` from `floor(available / cellSide)`:
  - Calculate leftover for each axis: `xWaste = availW - gridCols * cellSide`, `yWaste = availH - gridRows * cellSide`
  - If `xWaste > 0.6 * cellSide`, increment `gridCols` by 1
  - If `yWaste > 0.6 * cellSide`, increment `gridRows` by 1
  - Then recompute `cellSide` to fit the new grid dims evenly: `cellSide = min(availW / gridCols, availH / gridRows)` (cells stay square)
- **60% threshold** — conservative enough to avoid micro-cells, aggressive enough to catch the 82% waste case on 1024x600
- **Floor minimums still apply:** 3 cols / 2 rows minimum regardless of snap

### Cell gap and gutter handling
- **Minimal fixed gap between cells** — small uniform gap (e.g., 2-4px) between all cells, same in both axes (square gap pattern)
- **Cells and gap stay square** — cell+gap pattern is uniform, no rectangular gaps
- **Remaining gutter** after snap+gap absorbed into symmetric edge margins (`offsetX`/`offsetY`) — same pattern as today, just less waste
- Gap is subtracted from available space before cell sizing: `effectiveW = availW - (gridCols - 1) * gap`, then `cellSide = min(effectiveW / gridCols, effectiveH / gridRows)`

### Navbar height deduction
- **Grid accounts for navbar thickness** when computing available height — subtract `UiMetrics.navbarThick` from available vertical space before computing gridRows
- This prevents the bottom row from being partially hidden behind the navbar
- Phase 09 removed dock height deduction; this is different — the navbar is permanent and always visible on the home screen (unlike the dock which was being deleted)
- Deduction applies based on navbar edge: bottom/top → subtract from height, left/right → subtract from width

### Claude's Discretion
- Exact gap size (2px vs 4px — whatever looks right with the glass card styling)
- Whether auto-snap logic lives in DisplayInfo.cpp (C++) or HomeMenu.qml (QML) — either is fine, just keep it consistent with the existing grid frame ownership split
- How the navbar communicates page state to display dots (signal from WidgetGridModel or direct QML binding)
- Navbar dot layout implementation (inline in NavbarControl, separate component, or extension of the clock control)
- Whether `UiMetrics.marginPage` is still needed as edge padding or if the gutter-to-edge offset replaces it

</decisions>

<specifics>
## Specific Ideas

- "Page indicators are taking up too much usable space — would like to see them on either side of the clock in the navbar instead"
- "The clock should be the indicator of the page you're on. If on the home page and 3 pages exist, two dots to the right. If on the settings page, two dots to the left."
- "Maybe add a little more flexibility in the grid physical size formula so it tries to avoid losing too much space to the gutters. Right now I feel like I'm losing almost a whole row and column to being just a little bit off on size."
- Current 1024x600 display gives 7x3 grid — goal is to recover enough space for 8x4 or similar

</specifics>

- Grid frame computed in QML (HomeMenu.qml lines 21-28): `gridCols`, `gridRows`, `offsetX`, `offsetY` derived from `DisplayInfo.cellSide`
- Navbar floats at z=100, edge-configurable (left/right/top/bottom)
- `NavbarControl.qml` renders individual controls (volume/clock/brightness) — page dots would be siblings or children of the clock control

- `Shell.qml`: Navbar positioned here — may need to expose pageView state to navbar scope
- `WidgetGridModel`: Exposes `pageCount` — navbar needs to read this for dot count
