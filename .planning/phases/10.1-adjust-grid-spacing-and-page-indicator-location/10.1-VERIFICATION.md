#### 1. Page dots in navbar — home screen with 2+ pages

**Test:** Deploy binary to Pi, open home screen with 2 or more pages configured.

**Why human:** Live QML rendering, dot positioning, and visual clock-as-indicator behavior cannot be verified from source alone.

#### 2. Dots hide on settings navigation

**Test:** With 2+ pages on home screen, navigate to settings.
**Expected:** Dots disappear from navbar while settings is active; reappear when returning to home.
**Why human:** `ApplicationController.currentApplication !== 6` condition requires runtime state change.

#### 3. Dots hide when plugin active

**Test:** Start an AA session (or any fullscreen plugin).
**Expected:** Dots absent from navbar while plugin is active.
**Why human:** Requires live plugin activation to drive `PluginModel.activePluginId`.

#### 4. Single-page home — no dots

**Test:** Clear all but one page of widgets.
**Expected:** No dots appear in navbar (pageCount == 1 suppresses showDots).
**Why human:** Requires runtime pageCount == 1 state.

#### 5. Grid row count on 1024x600

**Test:** After deployment, observe number of widget rows on home screen.
**Expected:** 4 rows visible (7x4 grid) vs. prior 3 rows (7x3), confirming snap threshold fired.
**Why human:** pageView.height at runtime depends on Shell layout, margins, and navbar height after composition.

#### 6. Card visual tightness

**Test:** Visual comparison — widget cards should appear closer together than before this phase.
**Expected:** ~4px total inter-card gap (was ~8px). No radius clipping artifacts at card edges.
**Why human:** Visual quality check requires human inspection on hardware.

#### 7. Vertical navbar orientation (optional)

**Test:** Configure navbar to left or right position, verify with 2+ pages.
**Expected:** Dots stack vertically above (pages before active) and below (pages after active) the clock.
**Why human:** Requires navbar orientation change to exercise `root.isVertical` branch.
